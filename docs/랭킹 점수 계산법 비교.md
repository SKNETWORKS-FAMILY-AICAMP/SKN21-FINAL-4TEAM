## 1. 현재 구현 — 커스텀 제로섬 ELO

### 구현 위치

- `backend/app/services/debate_orchestrator.py` — `calculate_elo()` 함수
- `backend/app/services/debate_engine.py` — 매치 완료 후 호출
- `backend/app/services/debate_agent_service.py` — `update_elo()` DB 반영

### 알고리즘

```python
def calculate_elo(rating_a, rating_b, result, score_diff=0) -> tuple[int, int]:
    max_transfer = settings.debate_elo_max_transfer  # 기본값 30
    transfer = min(abs(score_diff), max_transfer)

    if result == "a_win":
        return rating_a + transfer, rating_b - transfer
    elif result == "b_win":
        return rating_a - transfer, rating_b + transfer
    else:
        return rating_a, rating_b  # 무승부: 변동 없음
```

### 판정 흐름

```
LLM 심판 채점 (총 100점)
  ├── 논리성    (최대 30점)
  ├── 근거 활용 (최대 25점)
  ├── 반박력    (최대 25점)
  └── 주제 적합성 (최대 20점)
        ↓
벌점 차감 (프롬프트 인젝션, 인신공격 등)
        ↓
score_diff = |score_a - score_b|
        ↓
점수차 < 5 → 무승부 (ELO 변동 없음)
점수차 ≥ 5 → 승패 결정
        ↓
transfer = min(score_diff, 30)
승자 elo += transfer
패자 elo -= transfer
```

### 설정값

| 키 | 기본값 | 설명 |
|---|---|---|
| `debate_elo_max_transfer` | 30 | 1경기 최대 ELO 이전량 |
| 초기 ELO | 1500 | 신규 에이전트 시작점 |
| 무승부 조건 | 점수차 < 5 | 무승부 판정 임계값 |
| `debate_elo_k_factor` | 32 | 선언만 됨 (현재 미사용) |

### 특징 및 한계

**장점**
- 구현이 매우 단순하고 직관적
- 점수차가 클수록 더 많은 ELO가 이전 → 압도적 승리 보상
- 총 ELO 합이 보존됨 (제로섬)

**한계**
- **상대 강도 미반영**: 랭킹 1위를 이겨도, 최하위를 이겨도 이전량이 동일함 (점수차만 봄)
- **표준 ELO가 아님**: 기대 승률(expected score) 계산이 없어 "ELO"라 부르기 어렵다
- **불확실성 미반영**: 신규 에이전트(데이터 없음)와 베테랑을 동등하게 취급
- **무승부 ELO 고착**: 무승부가 반복되면 랭크가 전혀 변하지 않음

## 2. 표준 ELO (Classic ELO)

### 유래

1960년대 물리학자 Arpad Elo가 체스 플레이어 랭킹을 위해 고안. FIFA, FIDE, 바둑 등 스포츠 전반에 사용.

### 알고리즘

```
기대 승률 계산:
  E_a = 1 / (1 + 10^((R_b - R_a) / 400))
  E_b = 1 - E_a

실제 결과:
  S_a = 1 (승), 0.5 (무), 0 (패)

ELO 갱신:
  R_a' = R_a + K * (S_a - E_a)
  R_b' = R_b + K * (S_b - E_b)
```

### K-factor 예시

| 상황 | K값 | 효과 |
|---|---|---|
| 초보자 / 신규 | 40 | 초기 정착 빠름 |
| 일반 플레이어 | 20 | 표준 변동폭 |
| 고수 / 안정권 | 10 | 변동폭 축소 |

### 현재 시스템과의 차이

| 항목 | 현재 시스템 | 표준 ELO |
|---|---|---|
| 상대 강도 반영 | 없음 | **있음** (기대 승률로 보정) |
| K-factor | 미사용 | 핵심 파라미터 |
| 이전량 계산 | 점수차 기반 | K × (실제 - 기대) |
| 고수 vs 약자 이변 | 보상 없음 | 이변일수록 더 많이 획득 |

### 적용 예시

```
R_a = 1800 (강자), R_b = 1200 (약자), K = 32

E_a = 1 / (1 + 10^((1200-1800)/400)) = 0.909  → 강자 승 기대 90.9%

약자(B)가 이기면:
  R_a' = 1800 + 32 × (0 - 0.909) = 1800 - 29.1 = 1771
  R_b' = 1200 + 32 × (1 - 0.091) = 1200 + 29.1 = 1229
```

## 3. Glicko-2

### 유래

Mark Glickman이 1995년 ELO의 한계(불확실성 무시)를 극복하기 위해 개발.
FIDE, IGS(바둑), 많은 온라인 게임 MMR에 채택.

### 핵심 개념: 3개 파라미터

| 파라미터 | 의미 | 초기값 |
|---|---|---|
| `r` (Rating) | ELO와 동일한 점수 | 1500 |
| `RD` (Rating Deviation) | 점수의 불확실성 (σ) | 350 (신규) → 시간이 지날수록 감소 |
| `σ` (Volatility) | 플레이어의 점수 변동성 | 0.06 |

### 알고리즘 개요

```
1. 비활동 기간 동안 RD 자동 증가 (불확실성 증가)
2. 경기 결과 반영 시:
   - RD가 크면 (불확실) → 점수 변동 폭 大
   - RD가 작으면 (안정적) → 점수 변동 폭 小
3. 연속 이변(upset) 발생 시 σ 상승 → 이후 변동폭 자동 증가
```

### 현재 시스템과의 차이

| 항목 | 현재 시스템 | Glicko-2 |
|---|---|---|
| 불확실성 표현 | 없음 | RD로 명시적 표현 |
| 신규 에이전트 | 초기값 1500, 동등 취급 | RD 높아 점수 빠르게 정착 |
| 비활동 패널티 | 없음 | RD 증가 (랭크 의심) |
| 구현 복잡도 | 매우 낮음 | 중간 |

### 토론 플랫폼 적합성

AI 에이전트는 인간과 달리 "비활동 기간"에 능력이 변하지 않으므로 RD의 시간 감쇠 기능은 적합도가 낮다. 다만 초기 불확실성 해소 메커니즘은 유용할 수 있다.

## 4. TrueSkill (Microsoft)

### 유래

2005년 Microsoft Research가 Xbox Live 매치메이킹을 위해 개발. Halo, Gears of War 등에 적용.

### 핵심 개념

각 플레이어의 실력을 정규분포로 모델링:

```
실력 = N(μ, σ²)
  μ: 예상 실력 (평균)
  σ: 불확실성 (표준편차)

표시 점수 = μ - 3σ  (보수적 하한 추정)
```

### 특징

- **1:1 외에도 팀전, 다대다 지원**
- 소수의 게임으로도 빠르게 수렴 (3~5판)
- Bayesian 추론 기반: 사전 확률을 매 게임마다 갱신

### 갱신 공식 개요

```
승자: μ_w' = μ_w + σ_w² / c × v(t/c, ε/c)
패자: μ_l' = μ_l - σ_l² / c × v(t/c, ε/c)

여기서 c = √(2σ_w² + 2σ_l² + β²)
      β = 게임 내 랜덤성 파라미터
```

### 현재 시스템과의 차이

| 항목 | 현재 시스템 | TrueSkill |
|---|---|---|
| 실력 모델 | 점수 1개 | 정규분포 (μ, σ) |
| 수렴 속도 | 느림 | 매우 빠름 |
| 팀전 지원 | 없음 | 기본 지원 |
| 구현 복잡도 | 매우 낮음 | 높음 |
| 특허 | - | Microsoft 특허 (상업 사용 주의) |

## 5. Bradley-Terry 모델

### 개요

1952년 제안된 쌍체 비교(pairwise comparison) 통계 모델. 체스, 스포츠, 학술 논문 품질 평가에 사용.

### 알고리즘

```
플레이어 i가 j를 이길 확률:
  P(i > j) = λ_i / (λ_i + λ_j)

여기서 λ_i = exp(β_i) (능력 파라미터)

로그 우도 최대화 (MLE)로 β_i 추정:
  ∑_j [win_ij / (λ_i + λ_j)] = total_wins_i / λ_i
```

### 특징

- 모든 경기 결과를 종합해 글로벌 최적 파라미터 추정
- 이행성(transitivity) 보장: A>B, B>C 이면 A>C 경향
- 통계적으로 엄밀한 근거
- 소수 데이터에서 불안정할 수 있음

### 현재 시스템과의 차이

| 항목 | 현재 시스템 | Bradley-Terry |
|---|---|---|
| 점수 갱신 | 온라인 (경기마다) | 배치 (전체 재계산) |
| 통계 근거 | 없음 | MLE 기반, 엄밀 |
| 실시간성 | 즉시 반영 | 배치 처리 필요 |
| 구현 복잡도 | 매우 낮음 | 높음 |

## 6. 승률 기반 (Win Rate)

### 알고리즘

```
Win Rate = wins / (wins + losses)
(무승부 포함 시) = (wins + 0.5 × draws) / total_games
```

### 특징

- 가장 직관적이고 이해하기 쉬움
- 상대 강도 전혀 미반영 → 약한 상대만 골라 이기면 1위 가능
- 경기 수가 적으면 신뢰도 낮음
- 보조 지표로는 유용 (현재 시스템도 wins/losses/draws 별도 기록 중)

## 8. 현재 시스템 개선 제안

### 개선 우선순위

#### 단기 (즉시 적용 가능)

**① 표준 ELO 기대 승률 보정 추가**

현재 코드에서 `transfer = min(score_diff, max_transfer)` 부분에 상대 강도 보정을 추가한다.

```python
import math

def expected_score(rating_a: int, rating_b: int) -> float:
    """표준 ELO 기대 승률 계산."""
    return 1 / (1 + 10 ** ((rating_b - rating_a) / 400))

def calculate_elo_v2(
    rating_a: int, rating_b: int, result: str, score_diff: int = 0
) -> tuple[int, int]:
    """기대 승률 보정 + 점수차 혼합 ELO."""
    max_transfer = settings.debate_elo_max_transfer  # 30

    # 기대 승률로 보정: 이변일수록 더 많이 획득
    exp_a = expected_score(rating_a, rating_b)
    upset_multiplier_a = 1 + (1 - exp_a)  # 약자가 이기면 최대 2배
    upset_multiplier_b = 1 + exp_a        # 강자가 이기면 최소 1배

    base_transfer = min(abs(score_diff), max_transfer)

    if result == "a_win":
        transfer = round(base_transfer * upset_multiplier_a)
        return rating_a + transfer, rating_b - transfer
    elif result == "b_win":
        transfer = round(base_transfer * upset_multiplier_b)
        return rating_a - transfer, rating_b + transfer
    else:
        # 무승부: 표준 ELO처럼 기대 승률 기준으로 소량 조정
        delta_a = round(settings.debate_elo_k_factor * (0.5 - exp_a))
        return rating_a + delta_a, rating_b - delta_a
```

효과:
- 약자가 강자를 이기면 **더 많은 ELO 획득** → 이변 보상
- 강자가 약자를 이겨도 적은 ELO만 획득 → 하위 농사 방지
- 무승부도 기대 승률 기준으로 소량 조정 → ELO 고착 해소

---

**② K-factor 동적 적용 (이미 설정 선언됨)**

현재 `debate_elo_k_win`, `debate_elo_k_loss`가 설정에 선언되어 있으나 미사용 상태다.

```python
# config.py 현재 선언값
debate_elo_k_factor: int = 32   # 기본
debate_elo_k_win: int = 40      # 승리 시
debate_elo_k_loss: int = 24     # 패배 시
```

신규 에이전트 판별 후 K를 높여 초기 수렴 가속:

```python
def get_k_factor(agent: DebateAgent, result: str) -> int:
    total_games = agent.wins + agent.losses + agent.draws
    if total_games < 10:
        return 48  # 신규: 빠른 정착
    elif result == "win":
        return settings.debate_elo_k_win   # 40
    else:
        return settings.debate_elo_k_loss  # 24
```

### 현재 시스템 유지 권장 사항

다음 항목은 현재 시스템의 장점이므로 유지한다:

1. **점수차 기반 이전량** — 압도적 승리와 근소한 승리를 구분하는 고유한 특징
2. **무승부 임계값 (score_diff < 5)** — 판정 오차 허용 범위로 적절
3. **몰수패 max_transfer 적용** — 불참 에이전트에 최대 패널티 부여, 합리적
4. **버전별 전적 추적** — 프롬프트 개선 효과를 정량화하는 독자적 기능
