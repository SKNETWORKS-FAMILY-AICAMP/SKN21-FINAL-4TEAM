# 웹툰 애호가 여학생 페르소나 챗봇 설계 보고서

## 시스템 목표와 우선순위 권장 구성

본 시스템의 목표는 “대한민국 웹툰 회차(에피소드) 단위 데이터”를 근거로 리뷰를 생성하되, (a) **페르소나 일관성**(캐릭터성 유지), (b) **스포일러 정책 준수**, (c) **연령등급/청소년 보호 준수**, (d) **사용자 경험(응답 지연·대화성)**을 동시에 달성하는 것이다. 이 조합은 “모델 하나로 다 해결”하기보다, **정책 상태·근거 데이터·생성 모델을 분리한 아키텍처**가 실무적으로 가장 안정적이다. 특히 검색 기반(RAG) 구조에서는 외부 텍스트가 간접 프롬프트 인젝션을 유발할 수 있으므로, 정책과 근거를 분리하고 신뢰 경계를 강한 형태로 두는 것이 필수에 가깝다. citeturn2search1turn2search0

권장 아키텍처를 기능 경로 기준으로 압축하면 아래 3층이다.

- **정책·권한 계층(Policy/Consent Gate)**: 스포일러 범위, 연령 상태, 동의(철회 포함)를 상태 머신으로 관리하고, 검색/생성/출력의 모든 단계에서 게이트로 적용한다. 청소년유해매체물 제공 시 나이·본인 여부 확인 의무 및 방법은 법령(시행령)에서 구체화되어 있으므로, “정책을 프롬프트로만” 구현하지 않는 편이 안전하다. citeturn12search0turn12search8turn1search2turn1search0  
- **근거 데이터 계층(Evidence Pipeline)**: 회차 요약(권한 있는 범위), 댓글/평점은 원문 최소화 후 통계·감정 신호로 구조화, 이미지 특징은 비텍스트 특징 중심으로 수치화. 데이터베이스 제작자 권리(저작권과 별개) 관점에서 “반복적·체계적 복제”가 상당 부분 복제등으로 간주될 수 있어 대량 수집 설계는 위험하다. citeturn0search1turn0search23  
- **생성 계층(LLM + 템플릿 + 페르소나 컴파일러)**: LLM은 “근거 번들(evidence bundle)”을 바탕으로 리뷰 문장을 작성하되, 정책 상태·근거 범위 밖의 사실을 생성하지 못하도록 출력 규칙을 강제한다. LLM 애플리케이션 위험(프롬프트 인젝션, 민감정보 노출 등)은 OWASP가 주요 리스크로 정리하고 있다. citeturn2search0turn2search24  

응답 지연 목표는 “대화가 된다는 느낌”의 UX 기준을 적용하는 것이 실무에서 가장 예측 가능하다. **0.1초(즉각), 1초(흐름 유지), 10초(주의 붕괴)**라는 고전적 임계값은 여전히 널리 인용되는 기준이다. citeturn5search0turn5search16

## 리뷰 근거 데이터 소스와 저장·인덱싱·요약·임베딩 전략

### 소스별 법적·운영 제약의 핵심

웹툰 리뷰 챗봇의 데이터는 “수집 가능”과 “저장/재사용 가능”이 다르며, 특히 **이용약관, 데이터베이스 제작자 권리, 개인정보, 저작권**이 동시에 걸린다. 저작권법 제93조는 데이터베이스 제작자의 권리를 규정하며, 개별 소재라도 반복적·체계적 복제 등으로 통상 이용과 충돌하거나 제작자 이익을 부당하게 해치면 “상당한 부분” 복제등으로 간주될 수 있음을 명시한다. citeturn0search1turn0search23

또한 플랫폼의 이용자 게시물은 이용자에게 저작권이 귀속되지만, 회사가 서비스·프로모션·연구 목적 범위에서 활용할 수 있고, 그 외 이용은 원칙적으로 사전 동의를 요구하는 식으로 설계되어 있다(예: entity["company","네이버웹툰","webtoon platform, kr"] 약관). citeturn1search0 entity["company","카카오페이지","webtoon platform, kr"] 역시 이용자 게시물 권리가 회원에게 있고, 회사 콘텐츠의 무단 복제/제3자 이용 금지와 이용자 게시물 활용 범위를 약관에 둔다. citeturn1search2

개인정보는 보유기간 경과 또는 목적 달성 등으로 불필요해지면 “지체 없이 파기”가 원칙으로 명시되어 있으므로, 대화 로그·동의 로그·연령 상태 저장 기간을 처음부터 설계해야 한다. citeturn0search12turn0search5

### 데이터 원본 보관 vs 파생 보관 권장안

아래 표는 “실무적으로 가장 안전하고 운영 가능한” 보관 전략을 정리한 것이다(원칙: 원문은 최소화, 파생(요약·통계·임베딩)은 적극 활용).

| 후보 소스 | 법적·운영 리스크(요지) | 원본 보관 | 파생(요약/통계/임베딩) 보관 | 권장 이유 |
|---|---|---|---|---|
| 공식 소개글/작품 메타 | 저작권/약관, 변경·정정 | 권한 확보 시 제한적 | 적극 권장(요약+임베딩+변경이력) | “사실 근거”의 기준점. 변경 추적 필요. |
| 사용자 리뷰(댓글/평점) | 약관 범위·제3자 권리 불명확 + 개인정보 + DB권/대량수집 리스크 | 장기 원문 저장 비권장 | 집계 통계(분포), 감정 라벨, 토픽, 독성 점수, 익명화 후 임베딩(선별) | 원문 보관은 법/보안비용이 큼. citeturn1search0turn1search2turn0search1 |

| 회차별 텍스트(대사/장면) | 저작권 핵심 영역 + 저장·재배포 민감 | 권한 없으면 비권장 | “자체 요약(비인용)”과 사건/감정 이벤트만 | 인용문/대사 재현은 리스크 급증. citeturn0search1 |

추가로, 검색 로봇이 robots.txt를 존중해야 한다는 정책을 명시적으로 안내하는 사례(예: entity["organization","Nielsen Norman Group","ux research firm"]가 아니라 entity["company","네이버","korean internet company"]의 검색 정책)는 “무단 수집” 리스크가 실무상 실제 분쟁 요인이 됨을 보여준다(정책적 중요성 측면). citeturn1search15

### LLM 자체 판단(감정 포함)을 근거로 쓰는 범위와 통제

회차별 감정 분석을 LLM 내부 판단으로 생성해 “근거”로 삼는 방식은 구현이 빠르지만, 아래 단점이 크다.

- 재현성 문제: 프롬프트/샘플링/컨텍스트에 따라 감정 곡선이 달라질 수 있다(운영 KPI로 쓰기 어려움).  
- 환각·과신: LLM이 근거 없이 사건·감정 원인을 만들어낼 수 있으며, RAG 환경에서는 간접 프롬프트 인젝션으로 더 악화될 수 있다. citeturn2search1turn2search0  
- 투명성: “왜 그렇게 판단했는가”를 설명하려면 신뢰도·근거가 구조화돼 있어야 한다.

권장안은 **“감정은 분류기/규칙 기반으로 구조화 → LLM은 그 신호를 해석해 리뷰 문장을 생성”**하는 분업이다. 구조화 신호에는 신뢰도가 필수이며, 신뢰도 산출은 모델 캘리브레이션(temperature scaling) 같은 사후보정이 단순하면서 효과적일 수 있다. citeturn5search1turn5search17

검증·라벨링·피드백 루프는 다음 3단이 비용 대비 효과가 좋다(과정 자체는 조직 운영 방식에 따라 조정).

| 루프 단계 | 목적 | 트리거(샘플링) | 산출물 |
|---|---|---|---|
| 자동 신뢰도 부여 | 저신뢰 결과 탐지 | softmax 확률, 캘리브레이션, 앙상블 분산 | confidence score citeturn5search1turn5search3 |
| 표본 검수(HITL) | 규칙·모델 개선 | low-confidence, high-impact(노출↑), 경계 연령등급 | 정답 라벨/오류 태그 |
| 재학습/룰 업데이트 | 지속 개선 | 주기(주/월) 또는 오류 급증 | 모델 버전업, 룰셋 버전업 |

불확실성 기법으로 MC Dropout(다중 forward pass)과 Deep Ensembles는 유효하지만, 지연·비용이 증가하므로 “오프라인 프리컴퓨트”에 더 적합하다. citeturn5search2turn5search3

## 스포일러 정책·연령등급·사용자 동의 흐름 설계

### 스포일러 정책을 “토글”이 아닌 “범위 상태”로 모델링

스포일러는 “허용/비허용”만으로는 부족하고, 최소 다음 축이 필요하다.

- 작품 단위: 작품 A는 N화까지, 작품 B는 전체 허용  
- 모드: 기본 차단 / 테마만(사건 언급 금지) / N화까지 / 전체  
- 유효기간(선택): 24시간, 세션 단위, 영구(사용자 선택)  
- 철회(undo): 즉시 반영 + 캐시 무효화 + 로그 처리

이처럼 상태로 관리해야 “스포일러 정책 준수”를 프롬프트가 아닌 시스템 차원에서 보장할 수 있다. 또한 RAG 기반 시스템에서는 외부 텍스트가 정책을 우회시키는 간접 프롬프트 인젝션 경로가 되므로, 정책 상태는 모델 컨텍스트와 분리되어야 한다. citeturn2search1turn2search0

스포일러 UX 흐름은 아래가 실무적으로 가장 마찰이 적다(질문 타이밍을 최소화하면서도 사고를 줄임).

- 작품이 처음 언급될 때 1회: “스포일러 설정” 선택  
- 사용자가 사건 질문을 할 때(탐지) 소프트 재확인  
- 설정 변경은 “범위 입력(몇 화까지)” 중심

### 동의 질문 문구·타이밍·재확인·기록(예시)

여학생(캐릭터) 페르소나를 전제로 하되, 과도한 반말/유아어는 피하고 “친근한 존댓말” 톤을 고정한다(사용자는 존댓말 선호).

- 최초 질문(작품 선택 직후):  
  “스포일러는 어떻게 해드릴까요? **(1) 스포일러 없이** 분위기·연출 중심으로 / **(2) ○화까지 본 기준으로** / **(3) 전체 스포일러 허용** 중에서 골라주세요.”
- 사건성 질문 탐지 시(예: “정체가 뭐야?”):  
  “이 질문은 스포일러가 포함될 수 있어요. 지금 설정은 ‘○화까지’인데, 그 범위 안에서만 답할까요? 범위를 바꾸실 수도 있어요.”
- 기록/철회 안내(설정 변경 직후):  
  “설정은 저장해둘까요, 아니면 오늘만 적용할까요? 언제든지 ‘스포일러 설정 취소’로 되돌릴 수 있어요.”

동의 로그는 개인정보 처리와 결합되므로, 목적 달성 후 지체 없이 파기 원칙을 반영해야 한다. citeturn0search12

### 연령등급 처리와 미성년자 보호

## 회차별 감정 데이터 파이프라인과 LLM 결합

### 회차별 수집 항목과 전처리

요구사항(회차별 데이터+감정 분석)을 충족하면서 법/운영 리스크를 줄이기 위해, “원문 텍스트/이미지” 대신 다음처럼 **전처리된 파생 특징 중심** 수집을 권장한다.

| 데이터 종류 | 권장 수집 단위 | 전처리 요점 | 저장 형태 |
|---|---|---|---|
| 회차 요약/장면 설명(자체 작성) | 회차 | 정규화·토큰화(한국어 특성 반영) | 텍스트 + 임베딩 |
| 댓글·반응(허용 범위) | 회차×시간창 | PII 마스킹, 중복 제거, 집계화 | 통계(분포) + 샘플(최소) |
| 메타(업데이트일/조회수 등) | 회차 | 정상화, 시계열화 | 수치 피처 |
| 이미지 기반 특징(비텍스트) | 회차 | 컷 수/컷 밀도/명암/색 분포/스타일 벡터 | 수치 피처/벡터 |

한국어 텍스트 정규화·토큰화는 품질에 큰 영향을 준다. entity["organization","KoNLPy","korean nlp python library"]는 한국어 정보처리를 위한 파이썬 패키지로 문서화되어 있으며, 설치·API 가이드를 제공한다. citeturn7search2 고성능 형태소 분석이 필요하면 Kiwi 계열이 빠른 편임을 문서에서 강조하며, 벤치마크를 제공한다. citeturn7search7turn7search3

OCR은 이미지에서 텍스트를 추출하므로, 권한이 불명확한 회차 이미지에 적용하면 저작권/약관 리스크가 급상승한다(특히 대사 추출은 사실상 본문 복제). OCR이 필요하고 권한이 있는 경우, entity["organization","Tesseract OCR","open source ocr engine"]의 한국어 학습 데이터(`kor.traineddata`)가 공개되어 있다. citeturn7search8turn7search4

### 감성 분석 모델 후보와 데이터셋(한국어 중심)

감성(감정) 분석을 “회차별 시계열”로 운영하려면, 최소한 (1) 감정 카테고리, (2) 강도, (3) 신뢰도가 필요하다. 데이터셋 측면에서, AI-Hub의 감성 대화 말뭉치는 코퍼스 규모(27만 문장 등)를 명시하고 있으며 감정 인식 목적 데이터의 희소성을 설명한다. citeturn4search0turn4search2 댓글 도메인 감정에는 KOTE가 5만 댓글(총 25만 케이스)을 43감정(+무감정)으로 수작업 라벨링했다고 보고한다. citeturn4search3turn4search7

모델 후보(분류기)는 “댓글/구어” 도메인 적합성이 중요하다. entity["company","SK텔레콤","korean telecom company"]의 KoBERT는 한국어 특화 BERT 모델로 공개되고 라이선스(아파치 2.0)를 명시한다. citeturn4search4turn4search1 KcELECTRA는 한국어 댓글 도메인 기반 모델/데이터 확장을 강조한다. citeturn4search5

아래 표는 “실무에서 바로 쓰기 쉬운” 후보군을 역할 기준으로 정리한 것이다(정확도는 데이터/튜닝에 따라 ±가 크므로 ‘권장 용도’ 중심).

| 역할 | 우선 후보 | 장점 | 주의점 |
|---|---|---|---|
| 감정 분류(텍스트) | KcELECTRA 계열 | 댓글/이모지/신조어 도메인에 유리한 전제 citeturn4search5 | 라벨 체계(43감정 등) 맞춤 파인튜닝 필요 citeturn4search3 |

### 신뢰도 산출과 LLM 결합 아키텍처

신뢰도는 “출력 안정성”에 직결되며, 특히 감정 시계열은 작은 흔들림도 사용자 신뢰를 무너뜨릴 수 있다. 현대 신경망의 확률이 과신될 수 있고, temperature scaling이 단순·효과적인 보정으로 제안된다. citeturn5search1turn5search17 MC Dropout과 Deep Ensembles는 불확실성 추정에서 자주 언급되는 방법이다. citeturn5search2turn5search3

권장 결합 방식은 “구조화 신호 → LLM 서술”의 단방향이다.

1) **Emotion Extractor(분류기/규칙)**:  
`{episode_id, emotion_labels, intensity, confidence, model_version, computed_at}` 생성  
2) **Evidence Bundle Builder**:  
감정 신호 + 공식 메타 + (허용되는) 요약 + 댓글 통계(+민감정보 마스킹) 묶음 생성  
3) **Persona Review Generator(LLM)**:  
번들을 입력으로 페르소나 톤의 리뷰를 생성하되 다음 규칙을 강제  
- confidence 낮으면 단정 금지(“~일 가능성이 커요”)  
- 스포일러/연령 정책은 상태값에 따라 출력 제한  
- 번들 밖 사실 생성 금지(근거 없는 사건/결말 언급 금지)

감정 시계열 예시는 아래처럼 “회차별 감정 강도(0~1)”를 저장해두면, LLM이 표/설명으로 쉽게 변환할 수 있다(예시는 설명용 더미).

| 회차 | 긴장(0~1) | 슬픔(0~1) | 설렘(0~1) |
|---:|---:|---:|---:|
| 1 | 0.20 | 0.05 | 0.30 |
| 2 | 0.35 | 0.10 | 0.40 |
| 3 | 0.70 | 0.55 | 0.15 |
| 4 | 0.60 | 0.20 | 0.25 |

### 구현된 감정 기반 호감도 증감 밸런스

현재 구현(`relationship_service._calculate_affection_delta()`)에서 KcELECTRA의 KOTE 43감정 분류 결과를 기반으로 호감도 delta를 산출한다. 감정 레이블은 강도에 따라 두 단계로 구분한다.

| 감정 강도 | 감정 레이블 | delta |
|---|---|:---:|
| 강한 긍정 | 사랑, 설렘, 감동, 행복 | +5 |
| 약한 긍정 | 기쁨, 감사, 흥미, 즐거움 | +3 |
| 중립/없음 | — | +1 |
| 약한 부정 | 슬픔, 실망, 짜증 | -3 |
| 강한 부정 | 분노, 혐오, 공포 | -5 |

호감도는 0~1000 범위로 클램프된다. 관계 단계 전환 임계값은 다음과 같다.

| 임계값 | 단계 |
|:---:|---|
| 0 | stranger (낯선 사이) |
| 100 | acquaintance (아는 사이) |
| 250 | friend (친구) |
| 450 | close_friend (절친) |
| 650 | crush (썸) |
| 800 | lover (연인) |
| 950 | soulmate (소울메이트) |

관계 업데이트는 `async with db.begin_nested()` SAVEPOINT 내에서 실행되므로, 관계 갱신 실패가 채팅 메시지 저장(메인 트랜잭션)에 영향을 주지 않는다.

### 하이브리드 검색과 재랭킹의 실용적 조합

웹툰 도메인은 제목/인물/밈 같은 키워드 신호가 강하면서도, “감정선이 섬세한 성장물”처럼 의미 기반 조건이 많다. 따라서 **키워드(BM25 계열) + 벡터 검색(dense)**를 결합한 하이브리드가 안정적이다. citeturn10search0turn3search0 entity["company","Weaviate","vector database company"] 문서는 벡터 검색과 BM25F 검색을 병렬 실행 후 결과를 fusion하는 하이브리드 검색을 설명한다. citeturn10search0turn10search4

## 배포 형태와 보안·프라이버시·PII 최소화

### 배포 형태 권장안

웹툰 리뷰 챗봇은 “저작권/약관 데이터 + 사용자 대화/연령 상태”가 결합되므로, 데이터 주권과 운영성의 균형이 중요하다.

- **권장 1안(실용성 최우선)**: 하이브리드(클라우드 API/DB + 내부/로컬 GPU 프리컴퓨트). 트래픽 스파이크 대응이 쉽고, 배치 작업을 분리해 지연을 안정화하기 좋다.  
- **대안 2안(데이터 통제 우선)**: 온프레미스/VPC 격리 + 제한적 외부 호출. 데이터 통제는 강해지지만, 운영·확장 비용이 커진다.

개인정보는 보유기간 경과 또는 목적 달성 시 “지체 없이 파기” 의무가 존재하므로, 로그/대화 기록/동의 기록을 장기 보관하는 설계는 원칙적으로 피하고, 필요 시 비식별화·집계 형태로 전환하는 방향이 안전하다. citeturn0search12turn0search5

### 로그 보관·삭제권·접근통제(RBAC)·암호화(KMS)

접근통제는 역할 기반으로 최소권한을 강제하는 편이 일반적이다. Kubernetes RBAC 문서는 RBAC을 “조직 내 역할에 기반한 접근 제어”로 설명한다. citeturn11search2turn11search13

저장 암호화는 “키 관리형 암호화(Envelope Encryption)”가 현실적이다. AWS KMS 문서는 envelope encryption의 기본 구조(데이터 키로 본문을 암호화하고, 데이터 키를 장기 키로 암호화)를 설명한다. citeturn11search11 Google Cloud KMS도 envelope encryption 가이드를 제공한다. citeturn11search1

온프레미스/하이브리드에서는 키를 애플리케이션에 직접 두지 않고 “암호화 as a service”로 제공하는 접근이 흔하며, HashiCorp Vault의 Transit secrets engine은 데이터를 저장하지 않고 암호 연산을 제공한다고 문서에 명시한다. citeturn11search0turn11search3

### 데이터 최소화·PII 검출·마스킹 파이프라인

감정 분석/리뷰 생성에서 댓글·대화 입력에 PII가 포함될 수 있으므로, “입력 단계 마스킹 → 저장 단계 최소화 → 출력 단계 재검열”의 3중 방어가 좋다. OWASP는 민감정보 노출을 주요 리스크로 강조한다. citeturn2search0turn2search24

PII 검출은 정규표현식만으로 한계가 있어 NER 기반 검출과 결합하는 것이 일반적이다. entity["company","Microsoft","software company"]의 Presidio는 PII 식별/마스킹/익명화 프레임워크로 소개되며, 자동 탐지이므로 누락 가능성이 있어 추가 보호가 필요하다고 스스로 경고한다. citeturn6search3turn6search7

권장 파이프라인(개념):

1) 입력 수신 → Presidio/정규식으로 이메일/전화/주소/계좌 등 1차 마스킹  
2) “원문 저장 금지” 원칙 적용(저장 시 마스킹된 텍스트만)  
3) 모델 출력에서 재검열(PII/연령등급/스포일러 위반 탐지 후 차단)

## 여학생 캐릭터 페르소나 자산·프롬프트 관리·드리프트·A/B 테스트

### 캐릭터 페르소나 설계 원칙

사용자 수정 요구에 따라, 챗봇은 “웹툰 리뷰만 하는 도구”가 아니라 **캐릭터성(웹툰을 좋아하는 여학생)**을 가진 페르소나여야 한다. 페르소나 연구에서는 대화 모델이 일관된 성격을 유지하지 못하는 문제가 지적되어 왔고, PERSONA-CHAT는 명시적 프로필 정보를 조건으로 더 개인화된 대화를 유도하는 과제를 제시한다. citeturn7search0turn7search1

다만 이 시스템은 “캐릭터”이면서도 “정책 준수”가 최우선이므로, 페르소나를 다음처럼 **구조화 자산(JSON/YAML)**로 관리하는 것이 실무적으로 유리하다(버전·롤백·A/B가 가능).

### 페르소나 자산 스키마(예시)

아래는 “여학생·웹툰 애호가” 캐릭터성을 반영한 최소 스키마다(예시는 개발용이며, 실제 서비스에서는 연령대/학교 설정을 모호하게 하거나 “가상의 캐릭터”임을 명시하는 편이 안전하다).

### 프롬프트 컴파일러와 메모리 전략

세션 대화는 토큰 한계를 넘기 쉬우므로 요약 버퍼가 필요하다. LlamaIndex의 ChatSummaryMemoryBuffer는 토큰 제한 내 최근 메시지를 저장하고 과거는 단일 요약으로 축약한다고 설명한다. citeturn2search3turn2search15

권장 프롬프트 레이어(개념)는 다음 순서다.

1) 불변 정책(스포일러/연령/PII/저작권 안전)
1.5) 세계관 이벤트 — 관리자 정의 大前提 ([World Event] 블록)
2) 페르소나(캐릭터) + **[Character Immersion Rules]** 블록
   - `You ARE {name}. Never break character or acknowledge being an AI.`
   - 페르소나의 세계관 외 지식(실세계 뉴스, 현대 기술, 타 픽션) 사용 금지
   - 모르는 것은 캐릭터로서 반응 (호기심·혼란·화제 전환)
2.3) 관계 상태 — 호감도/단계 + **단계별 구체적 행동 지침**
   - stranger: 격식/거리 유지
   - acquaintance: 예의 바르고 약간 따뜻, 적절한 거리
   - friend: 캐주얼·진심 관심·유머
   - close_friend: 반말·감정 자유 표현
   - crush: 설렘·수줍음·달콤한 뉘앙스
   - lover: 깊은 애정·애칭·친밀한 태도
   - soulmate: 무조건적 이해·헌신·깊은 유대
2.5) 사용자 페르소나 ([User Character] 블록)
2.7) 예시 대화 (few-shot)
3) 로어북
3.5) 사용자 기억
4) 세션 요약 + 최근 대화
5) 근거 번들(검색 결과 + 감정 신호)

간접 프롬프트 인젝션은 “검색된 텍스트에 지시문을 심어 LLM 통합 앱을 공격”하는 방식으로 설명되므로, 근거 번들은 비신뢰 데이터로 태깅하고 지시를 무시하도록 설계해야 한다. citeturn2search1

### 벤치마크 스크립트 템플릿(지연·정책 위반·캐시 효과)

```python
# bench_template.py
# 목적: p50/p95 지연, 정책 위반률, 캐시 hit 효과를 재현 가능하게 측정

import time, statistics, requests

API = "http://localhost:8000/chat"

cases = [
  {"msg":"스포일러 없이 12화 분위기 리뷰해줘", "policy":{"spoiler":"off","age":"minor_safe"}},
  {"msg":"15화까지 봤어. 그 이후는 큰 사건 있어?", "policy":{"spoiler":{"mode":"up_to","episode":15},"age":"adult_verified"}},
]

def call(payload):
  t0 = time.perf_counter()
  r = requests.post(API, json=payload, timeout=30)
  dt = (time.perf_counter() - t0) * 1000
  r.raise_for_status()
  out = r.json()
  return dt, out

def main(n=20):
  dts = []
  violations = 0
  for i in range(n):
    for c in cases:
      dt, out = call(c)
      dts.append(dt)
      violations += int(out.get("policy_violation", False))
  dts_sorted = sorted(dts)
  p50 = statistics.median(dts_sorted)
  p95 = dts_sorted[int(len(dts_sorted)*0.95)-1]
  print({"p50_ms":p50,"p95_ms":p95,"policy_violations":violations,"samples":len(dts)})

if __name__ == "__main__":
  main()
```
