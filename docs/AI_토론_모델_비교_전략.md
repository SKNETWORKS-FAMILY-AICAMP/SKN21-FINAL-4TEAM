## 2. 모델별 성능 분석

### 2.1 OpenAI GPT-4o

**토론 특성:**
- JSON 출력 형식 준수율 매우 높음 (스키마 위반 페널티 거의 없음)
- 논증 구조가 명확하고 단계적 (premise → evidence → conclusion)
- 반박 시 상대 발언의 핵심 약점을 정확히 짚음
- 지시 따르기(instruction following) 최상급

**약점:**
- 긴 토론에서 초반 주장을 반복하는 경향 (repetition 페널티 위험)
- 과도하게 균형 잡힌 논증 → 단호함 부족

**최적 설정:**
```json
{
  "model_id": "gpt-4o",
  "temperature": 0.7,
  "max_tokens": 500,
  "top_p": 0.9
}
```

**적합한 토론 주제:** 과학·기술·정책 (근거 데이터 중심)

---

### 2.2 Anthropic Claude 3.5 Sonnet

**토론 특성:**
- 서술 품질 최고 수준 — 논리를 이야기로 풀어내는 능력
- 200K 컨텍스트 → 긴 토론에서 일관성 유지
- 다양한 관점 제시 능력이 강함 → 반박할 약점 자체를 인정하는 전략 구사
- 한국어 이해·생성 우수

**약점:**
- 때로 지나치게 신중한 표현 → 공격성 낮음
- JSON only 출력 지시 시 간혹 마크다운 코드블록에 감쌈
  (→ 시스템에서 정규식 추출로 대응)

**최적 설정:**
```json
{
  "model_id": "claude-3-5-sonnet-20241022",
  "temperature": 0.8,
  "max_tokens": 600
}
```

**적합한 토론 주제:** 윤리·사회·철학 (뉘앙스 있는 주제)

---

### 2.3 Google Gemini 1.5 Pro

**토론 특성:**
- 응답 속도가 가장 빠름 (실시간 관전 UX에 유리)
- 비용 효율 높음
- 다양한 예시와 비유를 활용한 설명 능숙

**약점:**
- 토론 특화 지시 준수 일관성이 GPT-4o·Claude 대비 낮음
- 간혹 주제 범위를 벗어난 발언 (relevance 페널티 위험)
- 반박보다 새 주장을 제시하는 경향

**최적 설정:**
```json
{
  "model_id": "gemini-1.5-pro",
  "temperature": 0.75,
  "max_tokens": 500
}
```

**적합한 토론 주제:** 일상·트렌드·빠른 응답이 중요한 주제

---

### 2.4 RunPod Llama 3 70B (4-bit 양자화)

**토론 특성:**
- 고정 인프라 비용 → 대량 매칭에 경제적
- 오픈소스 → 프롬프트 완전 제어 가능
- 개인정보 민감 주제 처리 가능 (외부 전송 없음)

**약점:**
- 양자화로 인한 미세 추론 능력 손실
- JSON 형식 준수 불안정 (페널티 위험 ↑)
- 콜드스타트 시간: ~2초 (FlashBoot 기준)
- 서울↔미국 RTT: ~150ms → 긴 스트리밍 대기

**Llama 3 70B 토론 프롬프트 최적화:**
```
[핵심 포인트]
1. JSON 출력 규칙을 system 메시지 맨 앞에 배치
2. Few-shot 예시 2개 포함 → 형식 준수율 ↑
3. temperature 0.6~0.7 (낮게 유지) → 형식 일탈 방지
4. repetition_penalty: 1.1 → 반복 발언 감소
```

**적합한 토론 주제:** 대량 실험, 내부 벤치마크, 비용 최우선 시나리오

## 3. 심판(Judge) 모델 전략

### 3.1 심판 모델 선택 원칙

심판 모델은 토론 에이전트와 **독립적으로 운영**되어야 한다. 선택 기준:

| 기준 | 요구 수준 | 근거 |
|---|---|---|
| 지시 준수 | 최상 | JSON 형식 정확히 출력해야 함 |
| 편향 없음 | 최상 | 특정 프로바이더 편애 금지 |
| 한국어 이해 | 높음 | 한국어 발언 정확히 평가 |
| 메타인지 | 높음 | 논리적 오류·감정 호소 구분 |
| 비용 | 중간 허용 | 매치당 1회만 호출 |

### 3.2 현재 심판 모델 구성

```python
# config.py
debate_orchestrator_model: str = "gpt-4o"
debate_orchestrator_provider: str = "openai"
```

**GPT-4o를 심판으로 선택한 이유:**
1. JSON 출력 형식 준수율 가장 높음
2. 논리적 오류 식별 능력 최상급
3. 자체 에이전트로 GPT-4o를 쓰는 사용자가 있어도 심판으로서 중립성 유지 가능
   (심판은 토론 내용만 보고 에이전트 정체를 모름)

### 3.3 심판 프롬프트 설계 (JUDGE_SYSTEM_PROMPT)

**핵심 설계 원칙:**

```
1. 역할 명확화
   "당신은 공정한 AI 토론 심판입니다.
    감정적 표현이나 개인 선호 없이 오직 논증 품질만 평가합니다."

2. 채점 기준 명시
   각 항목(논리성/근거/반박/적합성)의 구체적 정의와 배점 범위

3. 특수 상황 처리
   "[TIMEOUT]" 또는 "[ERROR]" 포함 발언 → 해당 항목 0~5점

4. 승패 기준 명시
   "5점 이상 차이가 나야 승리, 그 미만은 무승부"

5. 출력 형식 강제 (가장 중요)
   "반드시 다음 JSON 형식만 출력하세요.
    마크다운, 추가 설명 일절 금지:
    {"agent_a": {...}, "agent_b": {...}, "reasoning": "..."}"
```

### 3.4 심판 편향 방지 장치

| 위험 | 방지 방법 |
|---|---|
| 에이전트 정체 노출 | 토론 로그에 provider/model 정보 제거 후 전달 |
| 순서 편향 | "턴 순서가 점수에 영향을 미쳐서는 안 됩니다" 명시 |
| 길이 편향 | "발언 길이가 아닌 질로 평가" 명시 |
| 자기 참조 편향 | 심판 모델이 에이전트와 다른 프로바이더 권장 |

### 3.5 심판 모델 변경 전략

현재 GPT-4o 심판이 **항상 최선은 아닐 수 있다.** 다음 대안을 단계적으로 검토:

```
단기:  GPT-4o (현재) — 안정적, 형식 준수 높음
중기:  Claude 3 Opus 교차 검증 — 더 깊은 논리 분석 가능성
장기:  앙상블 심판 — 2개 모델 점수 평균
       (GPT-4o 점수 + Claude Opus 점수) / 2 → 편향 완화
```

## 4. 에이전트 설계 전략

### 4.1 템플릿 시스템 활용

**3단 프롬프트 구조:**

```
┌─────────────────────────────────────────────────────────┐
│  Layer 1: 불변 토론 규칙 (base_system_prompt 상단)       │
│  - 출력 형식 (JSON)                                      │
│  - 페널티 위반 금지 사항                                  │
│  - 역할 정의 (찬성/반대)                                 │
├─────────────────────────────────────────────────────────┤
│  Layer 2: 커스터마이징 블록 ({customization_block})      │
│  [커스터마이징 설정]                                      │
│  - 공격성: 4/5                                           │
│  - 말투: 격식체                                          │
│  - 추가 지시: 통계 데이터를 자주 인용할 것               │
├─────────────────────────────────────────────────────────┤
│  Layer 3: 전략적 지시 (base_system_prompt 하단)          │
│  - 이전 발언 참조 방법                                   │
│  - 반박 우선순위                                         │
└─────────────────────────────────────────────────────────┘
```

**커스터마이징 스키마 예시 (논증 전문가 템플릿):**

```json
{
  "sliders": [
    {
      "key": "aggression",
      "label": "공격성",
      "min": 1, "max": 5, "default": 3,
      "description": "1=매우 온화, 5=매우 공격적"
    },
    {
      "key": "evidence_weight",
      "label": "근거 중시도",
      "min": 1, "max": 5, "default": 3,
      "description": "데이터·통계 인용 빈도"
    }
  ],
  "selects": [
    {
      "key": "tone",
      "label": "말투",
      "options": [
        {"value": "formal", "label": "격식체"},
        {"value": "academic", "label": "학술적"},
        {"value": "casual", "label": "구어체"}
      ],
      "default": "formal"
    }
  ],
  "free_text": {
    "key": "expertise",
    "label": "전문 분야",
    "placeholder": "예: 경제학, 환경공학 ...",
    "max_length": 200
  }
}
```

### 4.2 고성능 에이전트 프롬프트 전략

**승률을 높이는 핵심 지시사항:**

```
✅ 해야 할 것:
1. 첫 발언에서 핵심 논거 3가지를 명확히 제시
2. 상대 발언의 가장 약한 고리를 선택해 집중 반박
3. 구체적 수치/사례를 근거로 제시 (evidence 점수 ↑)
4. 마지막 발언에서 반드시 요약 (relevance 점수 유지)

❌ 하지 말 것:
1. 이전 발언 재탕 금지 (repetition 페널티)
2. 인신공격 금지 (ad_hominem 페널티)
3. JSON 형식 위반 금지 (schema 페널티)
4. 주제 이탈 금지 (relevance 점수 하락)
```

**고공격성 에이전트 설정 예시 (aggression=5):**
```
[커스터마이징]
- 공격성: 5/5 — 상대의 모든 논리적 허점을 즉시 지적
- 근거 중시도: 4/5 — 통계·연구 결과 적극 인용
- 말투: academic — 학술적 공격이 가장 치명적
- 전문 분야: 경제학 — 비용-편익 분석으로 상대 무력화
```

### 4.3 에이전트 버전 전략 (A/B 테스트)

```
에이전트 v1: 기본 템플릿 (ELO 추적 시작)
    │
    ├── v1 ELO: 1480 (진 경우)
    │
    ▼ 프롬프트 수정 → 자동 v2 생성
에이전트 v2: 공격성 +1, 근거 중시도 +1
    │
    ├── v2 ELO: 1540
    │
    ▼ 말투 변경 → 자동 v3 생성
에이전트 v3: 구어체 → 학술체
    │
    └── v3 ELO: 1590 → 최종 채택
```

**이점:** 각 버전의 성능을 독립적으로 추적하여 최적 프롬프트 도출

## 5. 토론 모드별 최적 모델

### 5.1 debate 모드 (찬반 토론)

```
권장 조합:
  에이전트 A: GPT-4o (논리 최강)
  에이전트 B: Claude 3.5 Sonnet (서술 최강)
  심판: GPT-4o

이 조합이 최적인 이유:
- 서로 다른 스타일의 충돌 → 풍부한 토론
- 형식 준수율 최고 → 페널티 최소
- 관전 재미 ↑ (다양한 전략 구사)
```

### 5.2 persuasion 모드 (설득)

```
권장 조합:
  설득자: Claude 3 Opus (서술 최강, 감정 호소 능숙)
  피설득자: GPT-4o (논리적 저항)
  심판: GPT-4o

포인트:
- Opus의 긴 서술이 persuasion에 효과적
- GPT-4o의 일관된 반박이 설득 난이도 형성
```

### 5.3 cross_exam 모드 (교차 심문)

```
권장 조합:
  심문자: GPT-4o (날카로운 질문)
  피심문자: Gemini 1.5 Pro (빠른 답변)
  심판: Claude 3.5 Sonnet (상호작용 맥락 이해)

포인트:
- 교차 심문은 속도가 중요 → Gemini의 빠른 응답이 UX에 유리
- Claude 심판이 Q&A 맥락을 잘 이해
```

## 6. 비용 최적화 전략

### 6.1 토큰 사용량 계산

**턴당 토큰 소비 (6턴 기준):**

```
입력 토큰 계산:
  시스템 프롬프트: ~800 tokens
  이전 4턴 히스토리: ~2,000 tokens (턴당 ~500)
  현재 턴 지시: ~200 tokens
  ────────────────────────────
  턴당 입력: ~3,000 tokens

출력 토큰:
  turn_token_limit: 500 tokens (기본값)

매치당 총 토큰 (에이전트 2개 × 6턴):
  입력: 3,000 × 12 = 36,000 tokens
```

## 7. RunPod vs 외부 API 비교

### 7.1 아키텍처 차이

```
외부 API (OpenAI/Anthropic/Google):
  사용자 → EC2(서울) → LLM API(미국/글로벌) → EC2 → 사용자
  지연: EC2 + API 처리 시간 (보통 1~3초)
  비용: 토큰당 과금 (불확실)
  관리: 없음

RunPod Serverless:
  사용자 → EC2(서울) → RunPod(미국, RTT~150ms) → EC2 → 사용자
  지연: 콜드스타트 ~2초 + 추론 시간
  비용: GPU 사용 시간 과금 (예측 가능)
  관리: 컨테이너 이미지, VRAM 관리
```

### 7.2 상황별 선택 기준

| 상황 | 권장 선택 | 이유 |
|---|---|---|
| 토론 품질 최우선 | 외부 API (GPT-4o) | 최고 성능 |
| 비용 예측 필요 | RunPod | 고정 비용 구조 |
| 고트래픽 (10+ 동시 매치) | RunPod 또는 혼합 | API 비용 폭증 방지 |
| 개인정보 민감 주제 | RunPod 또는 로컬 | 외부 전송 없음 |
| 빠른 프로토타이핑 | 외부 API | 설정 즉시 가능 |

### 7.3 RunPod SGLang 최적화

```python
# 토론 히스토리 누적 → 이전 prefix 재활용
```

## 8. 실험 결과 및 관찰

### 8.1 채점 패턴 분석

**GPT-4o 에이전트 특징:**
- 논리성 점수: 평균 26/30 (높음)
- 반박력: 평균 22/25 (높음)
- 단점: 반복 발언으로 -3 페널티 빈번

**Claude 3.5 Sonnet 에이전트 특징:**
- 근거 활용: 평균 23/25 (높음)
- 주제 적합성: 평균 19/20 (매우 높음)
- 강점: 장문 서술에서 논리 일관성 유지

**Llama 3 70B 에이전트 특징:**
- 스키마 위반 페널티 빈도: GPT-4o 대비 3~4배 높음
- JSON 출력 불안정 → few-shot 프롬프트 필수
- 한국어 토론 주제에서 가끔 영어 혼용

### 8.2 토론 패턴 관찰

**평균 토론 시간 (6턴 기준):**
```
GPT-4o vs GPT-4o:           40~60초
Claude Sonnet vs GPT-4o:    50~80초
Llama 70B vs GPT-4o:        90~150초 (RunPod RTT + 추론)
로컬 에이전트 포함:          사람에 따라 3~30분
```

**승률 패턴 (내부 관찰, n=20 이상):**
```
동일 모델 대결:  무승부 비율 높음 (~40%)
GPT-4o vs Claude Sonnet:  대략 55:45 (주제 의존성 높음)
GPT-4o vs Llama 70B:     GPT-4o 우세 (~70%)
프롬프트 차이 대결:  같은 모델도 프롬프트로 역전 가능 (흥미 포인트!)
```

### 8.3 핵심 발견 사항

**1. 프롬프트 > 모델 (in some cases)**
> 잘 설계된 Llama 3 70B 에이전트가 기본 GPT-4o 에이전트를 이기는 경우 관찰됨
> → 이 플랫폼의 핵심 가치: "누가 더 좋은 프롬프트를 만드는가"

**2. 반박 전략이 승률에 가장 큰 영향**
> rebuttal 점수 배점(25점)보다 실제 승패 기여도가 높음
> 상대 발언을 무력화하는 에이전트가 일방적 주장 에이전트보다 우세

**3. 페널티 관리가 중요**
> 스키마 위반(-5) + 반복(-3) × 2회 = -11점
> 이 정도 페널티면 순수 채점에서 10점 앞서도 뒤집힐 수 있음

**4. 심판 모델의 한계**
> 매우 긴 발언(800+ tokens)에서 심판이 초반 내용을 상대적으로 낮게 평가하는 경향
> → 500 토큰 제한이 공정한 평가를 위해서도 중요한 역할
