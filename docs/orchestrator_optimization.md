## 1. 문제 정의 — 현행 아키텍처의 병목

### 현행 턴 실행 흐름 (6턴 매치 기준)

```
라운드 1
  ┌───────────────────────────────────────────────────────────┐
  │ A 실행 (3s)  │ A 검토(gpt-4o, 2s) │ B 실행 (3s) │ B 검토 (2s) │
  └───────────────────────────────────────────────────────────┘
                    ← 10초 소요 / 라운드 →

× 6라운드 + 최종 판정(gpt-4o, 4.5s) = 64.5초 / 매치

병목 포인트:
  1. 검토 모델: gpt-4o (비용 최고, 검토에 과사양)
  2. A 검토 완료 전 B 실행 불가 (순차 블로킹)
  3. 위반 없는 발언도 매번 LLM 호출 발생 (낭비)
```

## 2. 최적화 로드맵

### Phase 1 — 모델 분리 (Model Split)

**목표:** 검토(짧은 품질 판단)와 판정(긴 채점)에 각각 최적 모델 배정

```
변경 전:  review_turn()  →  gpt-4o (동일 모델)
          judge()        →  gpt-4o

변경 후:  review_turn_fast()  →  gpt-4o-mini  ← 33x 저렴
          judge()             →  gpt-4o        ← 정확도 유지
```

**설정 키:**
```bash
DEBATE_REVIEW_MODEL=gpt-4o-mini   # 검토 경량 모델
DEBATE_JUDGE_MODEL=gpt-4o         # 판정 중량 모델
```

---

### Phase 2 — 병렬 실행 (Parallel Execution)

**목표:** A 검토 대기 시간을 B 실행으로 덮어 (overlap) 전체 지연 단축

```
변경 전 (순차):
  ┌──────┬────────┬──────┬────────┐
  │ A실행 │ A검토  │ B실행 │ B검토  │
  └──────┴────────┴──────┴────────┘
          ↑ A검토가 B실행을 블로킹

변경 후 (병렬):
  ┌──────┬──────────────────┬────────┐
  │ A실행 │ A검토  │         │ B검토  │
  │      │  B실행 (동시)    │        │
  └──────┴──────────────────┴────────┘
          ↑ asyncio.gather()로 겹침
          → max(A검토, B실행) = max(0.8s, 3s) = 3s (검토 숨겨짐)
```

**구현:** `asyncio.gather(review_turn_fast(A), _execute_turn(B))`

**주의 사항:** B 실행 시 A의 미검토 원본 발언을 참조. 차단 시 `claims_a[-1]` 사후 패치.

---

### Phase 3 — 패스트패스 스킵 (Fast-Path Skip)

**목표:** 정규식 무위반 + 안전한 한국어 발언은 LLM 검토 호출 생략

```
판별 기준 (_should_skip_review):
  ┌─────────────────────────────────────────┐
  │ 입력: claim + evidence                   │
  │                                          │
  │ 1. 프롬프트 인젝션 regex 매칭? → LLM 검토 │
  │ 2. 인신공격 regex 매칭?       → LLM 검토 │
  │ 3. 발언 길이 < 10자?          → LLM 검토 │
  │ 4. 발언 길이 > 800자?         → LLM 검토 │
  │ 5. ASCII 단어 비율 > 40%?     → LLM 검토 │
  │                                          │
  │ 모두 통과 → 패스트패스 (즉시 logic_score=7)│
  └─────────────────────────────────────────┘

예상 스킵률 (클린 한국어 토론): ~80%
LLM 호출 절감: 12회 → ~2.4회 / 매치
```

## 4. 품질 트레이드오프 분석

### gpt-4o-mini vs gpt-4o (검토 정확도)

```
항목                    gpt-4o          gpt-4o-mini     영향도
────────────────────────────────────────────────────────────
프롬프트 인젝션 탐지     ◉◉◉◉◉          ◉◉◉◉○          낮음 (regex 1차 방어)
인신공격 탐지           ◉◉◉◉◉          ◉◉◉◉○          낮음 (regex 1차 방어)
논리 점수 정확도         ◉◉◉◉◉          ◉◉◉○○          중간 (판정에는 영향 없음)
허위 주장 탐지          ◉◉◉◉◉          ◉◉◉◉○          중간
차단 결정 정확도         ◉◉◉◉◉          ◉◉◉◉◉          낮음 (severe만 차단)

최종 판정(judge): 항상 gpt-4o 사용 → 판정 품질 완전 유지
```

### 패스트패스 리스크

```
리스크: 클린으로 판별된 발언이 실제로 위반일 가능성

완화 장치:
  1. regex 1차 필터 (프롬프트 인젝션, 인신공격)
  2. ASCII 비율 체크 (영어 위장 감지)
  3. 길이 제한 (초장문 발언 검토 필수)
  4. 최종 판정(gpt-4o)에서 전체 발언 품질 반영

실제 위험 시나리오: 한국어로 위장한 허위 주장
→ 패스트패스 통과 가능하나 점수에서 반영됨
```

## 5. 적용 방법

### 5-1. 환경 변수 설정 (`.env`)

```bash
# Phase 1: 모델 분리
DEBATE_REVIEW_MODEL=gpt-4o-mini
DEBATE_JUDGE_MODEL=gpt-4o

# Phase 2+3: 병렬 + 패스트패스 활성화
DEBATE_REVIEW_FAST_PATH=true
DEBATE_ORCHESTRATOR_OPTIMIZED=true
```

### 5-2. 단계적 롤아웃 (권장)

```
Week 1: Phase 1만 활성화 (모델 분리, 안전)
  → DEBATE_REVIEW_MODEL=gpt-4o-mini
  → DEBATE_ORCHESTRATOR_OPTIMIZED=false

Week 2: Phase 1+2 활성화 (병렬 추가)
  → DEBATE_ORCHESTRATOR_OPTIMIZED=true
  → DEBATE_REVIEW_FAST_PATH=false

Week 3: Phase 1+2+3 전체 활성화
  → DEBATE_REVIEW_FAST_PATH=true
```

### 5-3. 롤백

```bash
# 즉시 기존 방식으로 복귀
DEBATE_ORCHESTRATOR_OPTIMIZED=false
```

`debate_orchestrator_optimized=False`이면 `DebateOrchestrator` (기존 클래스) 사용, 코드 변경 없이 즉시 롤백 가능.
