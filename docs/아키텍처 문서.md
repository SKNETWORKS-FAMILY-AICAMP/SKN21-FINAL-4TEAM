## 1. 아키텍처 개요

### 1.1 설계 철학

**하이브리드 스플릿 브레인** — 제어 계층(EC2 서울)과 GPU 추론 계층(RunPod US Serverless)을 분리하여 비용 효율과 성능을 동시에 확보한다.

**3층 분리 원칙:**

```
┌─────────────────────────────────────────────────────────────┐
│  1층: 정책/권한 계층                                         │
│  스포일러·연령·동의·RBAC를 DB 상태 머신 + 코드 게이트로 강제  │
│  (프롬프트 의존 금지)                                        │
├─────────────────────────────────────────────────────────────┤
│  2층: 근거 데이터 계층                                       │
│  회차 요약·감정 신호·댓글 통계·로어북을 구조화 저장            │
│  (배치 사전 계산 우선)                                       │
├─────────────────────────────────────────────────────────────┤
│  3층: 생성 계층                                              │
│  사용자 선택 LLM이 근거 번들 + 페르소나 기반으로 응답 생성    │
│  (토큰 사용량 기록 필수)                                     │
└─────────────────────────────────────────────────────────────┘
```

## 2. 데이터 흐름

### 2.1 채팅 메시지 흐름

```
사용자 입력
    │
    ▼
[1] PII 마스킹 (Presidio)
    │ 개인정보 감지 → 마스킹 후 저장
    ▼
[2] 정책 게이트 검증
    │ 스포일러 범위, 연령등급, 동의 상태 확인
    │ 위반 시 → 403 즉시 반환
    ▼
[3] 크레딧/할당량 검증
    │ credit_balance 확인 + quota 한도 확인
    │ 부족 시 → 429 반환
    ▼
[4] 프롬프트 컴파일 (8-layer)
    │ ① 불변 정책
    │ ② 페르소나 시스템 프롬프트
    │ ③ 관계 상태 (호감도/단계)
    │ ④ 사용자 페르소나
    │ ⑤ 예시 대화
    │ ⑥ 사용자 기억
    │ ⑦ 세션 요약 + 최근 대화
    │ ⑧ 근거 번들 (RAG 검색 + 감정 신호)
    ▼
[5] LLM 추론 (inference_client)
    │ llm_models 테이블 → provider별 분기
    │ SSE 스트리밍으로 청크 전송
    ▼
[6] 후처리
    │ ├─ 감정 분석 (KcELECTRA) → emotion_signal
    │ ├─ 토큰 사용량 기록 → token_usage_logs
    │ ├─ 크레딧 차감 → credit_ledger
    │ ├─ 호감도 갱신 → persona_relationships
    │ └─ 메모리 추출 (5턴마다) → user_memories
    ▼
[7] 응답 저장 → chat_messages
```

### 2.2 LLM 모델 라우팅

```
요청 → inference_client.generate() / generate_stream()
         │
         ├─ session.llm_model_id 또는 user.preferred_llm_model_id
         │
         ├─ llm_models 테이블에서 provider/model_id/endpoint 조회
         │
         ├─ provider별 분기:
         │   ├─ 'runpod'    → RunPod Serverless API (SGLang, OpenAI-compatible)
         │   ├─ 'openai'    → OpenAI API (GPT-4o 등)
         │   ├─ 'anthropic' → Anthropic API (Claude 등)
         │   └─ 'google'    → Google Gemini API
         │
         ├─ Langfuse 트레이싱 (generation 기록)
         ├─ Prometheus 메트릭 기록
         │
         └─ 응답 + 토큰 수 → token_usage_logs INSERT
```

### 2.3 TTS 합성 흐름

```
사용자가 TTS 버튼 클릭
    │
    ▼
POST /api/tts/synthesize-message
    │ { message_id, voice?, speed? }
    ▼
[1] 메시지 소유권 확인
    │ chat_messages → chat_sessions → user_id 검증
    ▼
[2] TTS Provider 분기 (settings.tts_provider)
    │ ├─ 'openai'     → OpenAI TTS API (tts-1 모델)
    │ └─ 'elevenlabs' → ElevenLabs API (multilingual_v2)
    ▼
[3] 오디오 파일 저장
    │ uploads/audio/{uuid}.mp3
    ▼
[4] 응답 → { audio_url, characters_count, provider, voice }
    │
    ▼
프론트엔드 Audio API로 재생
```

### 2.4 이미지 생성 흐름

```
POST /api/image-gen/generate
    │ { prompt, style?, width?, height?, negative_prompt?, seed? }
    ▼
[1] 스타일 프리셋 적용
    │ anime / realistic / cartoon / oil_painting / watercolor / pixel
    │ → 프롬프트 접두사 추가
    ▼
[2] Provider 분기 (settings.image_gen_provider)
    │ ├─ 'openai'    → DALL-E 3 API (b64_json 응답)
    │ └─ 'stability' → Stability AI SD3 API (바이너리 응답)
    ▼
[3] 이미지 파일 저장
    │ uploads/images/generated/{uuid}.png
    ▼
[4] 응답 → { image_url, prompt, style, provider }
```

## 4. 보안 아키텍처

### 4.1 인증/인가

```
┌─ 인증 흐름 ──────────────────────────────────────────────────────┐
│                                                                    │
│  POST /api/auth/login                                              │
│    │ nickname + password_hash (bcrypt)                              │
│    ▼                                                                │
│  JWT 발급 (HS256, SECRET_KEY 서명)                                 │
│    │ payload: { sub: user_id, role, exp }                           │
│    ▼                                                                │
│  클라이언트: localStorage.setItem('token', jwt)                     │
│    │                                                                │
│  이후 모든 API 요청:                                               │
│    Authorization: Bearer {jwt}                                     │
│    │                                                                │
│  미들웨어 검증:                                                    │
│    ├─ decode_access_token() → payload 추출                         │
│    ├─ is_token_blacklisted() → 로그아웃 토큰 차단                  │
│    └─ DB에서 user 조회 → banned_until 확인                         │
└────────────────────────────────────────────────────────────────────┘
```

### 4.2 RBAC (역할 기반 접근 통제)

| 역할 | 접근 범위 | 의존성 함수 |
|---|---|---|
| user | 사용자 API (91개) | `get_current_user` |
| admin | + 관리자 API (36개) | `require_admin` |
| superadmin | + 파괴적 작업 | `require_superadmin` |
| adult_verified | + 18+ 콘텐츠 | `require_adult_verified` |

### 4.3 연령등급 게이트

```
API 요청 (페르소나/채팅 접근)
    │
    ▼
persona.age_rating vs user.age_group 교차 검증
    │
    ├─ age_rating='all'  → 모든 사용자 허용
    ├─ age_rating='15+'  → minor_safe, adult_verified 허용
    └─ age_rating='18+'  → adult_verified만 허용
                             │
                             └─ 미인증 시 → 403 AUTH_ADULT_REQUIRED
```

### 4.4 Rate Limiting

| 엔드포인트 | 제한 | 윈도우 |
|---|---|---|
| `/api/auth/*` | 20 req | 60초 |
| `/api/chat/*` | 30 req | 60초 |
| `/api/*` (일반) | 60 req | 60초 |
| `/api/admin/*` | 120 req | 60초 |

구현: Redis 슬라이딩 윈도우 (`core/rate_limit.py`)

## 부록 A: 하이브리드 아키텍처 설계 근거

> 이전 `EC2_RunPod 프로토타입 구현 최적화 방안.md` 문서를 통합 정리한 내용.

### A.1 RP 챗봇의 기술적 특수성

일반적인 Q&A 봇과 달리, RP 챗봇은 다음과 같은 기술적 특수성을 가진다:

1. **긴 컨텍스트 윈도우:** RP 세션은 수백 턴 이상 지속될 수 있으며, 캐릭터 설정(Persona), 세계관 정보(Lorebook), 과거 대화 내역이 지속적으로 프롬프트에 포함되어야 한다. KV 캐시 관리 효율성이 성능의 핵심이다.
2. **상태 유지(Statefulness):** 사용자의 선택이나 주사위 굴림 결과에 따라 스토리 분기가 달라지며, 게임 로직의 통합을 요구한다.
3. **지연 시간 민감성:** 실시간 대화의 몰입감을 위해 생성 속도(Tokens/s)와 첫 토큰 생성 시간(TTFT)이 중요하지만, 비용 절감을 위해 물리적으로 먼 GPU 리전을 사용해야 하는 상충 관계가 존재한다.

### A.2 SGLang vs vLLM: 추론 엔진 선정 근거

현재 가장 널리 사용되는 vLLM은 PagedAttention으로 메모리 효율성을 극대화했지만, 멀티 턴 대화에서는 SGLang의 **RadixAttention**이 월등하다.

| 항목 | vLLM | SGLang |
|---|---|---|
| KV 캐시 구조 | LRU 리스트 | 라딕스 트리 |
| 접두사 캐싱 | 블록 단위 정확 일치 (배치 최적화) | 부분 일치 동적 탐색 (대화형 최적화) |
| 멀티 턴 성능 | 기준 | **10–20% 처리량 향상** |
| RP 시나리오 적합성 | 보통 | **우수** |

**RadixAttention 작동 원리:**
- 시스템 프롬프트 + 이전 대화가 '접두사(Prefix)'가 되어 라딕스 트리에서 자동으로 KV 캐시 재사용
- 두 번째 턴부터 프리필(Prefill) 단계 연산 비용이 거의 0에 수렴
- 초당 과금되는 서버리스 환경에서 직접적인 비용 절감

### A.3 Serverless vs Dedicated Pod: 비용 분석

**사용량 시나리오:**
- 동시 접속자 최대 10명, 일일 총 1,000회 메시지
- 메시지당 평균 처리 시간 5초, 모델 Llama 3 70B, GPU A100 80GB

| 방식 | 과금 단위 | 일일 GPU 가동 | 일일 비용 | **월간 비용** |
|---|---|---|---|---|
| **Serverless** | 초 단위 | ~1.4시간 (5,000초) | $3.80 | **$114** |
| Dedicated Pod | 시간 단위 | 24시간 | $35.76 | **$1,073** |

**손익분기점:** 하루 ~13시간 이상 GPU가 연속 가동되는 상황이 아니면 Serverless가 압도적으로 유리. 현재 시나리오에서 **약 90% 비용 절감**.

### A.4 네트워크 지연 극복: SSE 스트리밍 전략

서울↔미국 RTT ~150ms를 체감 지연에서 감추기 위한 SSE(Server-Sent Events) 전략:

- **SSE vs WebSockets:** 단방향 텍스트 스트리밍에는 SSE가 가볍고 구현이 용이. HTTP 프로토콜 그대로 사용하여 방화벽 문제 자유
- **구현 흐름:** FastAPI → RunPod (`stream=True`) → AsyncGenerator → SSE 이벤트 → 클라이언트 '타자기 효과'로 첫 토큰부터 렌더링

### A.5 콜드 스타트 관리: FlashBoot

- **RunPod FlashBoot:** Docker 이미지 레이어 고속 복원으로 컨테이너 기동 ~2초 이내
- **Active Worker 전략:** 프로토타입 단계에서는 `min_workers=0` (비용 최적화), 트래픽 집중 시간대에 Scheduled Ping으로 워커 웜업 가능

### A.6 게임 메카닉: Function Calling (주사위)

LLM은 확률적 텍스트 생성기이므로 공정한 주사위 결과를 생성할 수 없다. EC2 애플리케이션 서버에서 Python `random` 모듈로 실제 난수를 생성한 후 결과를 LLM 컨텍스트에 재입력하여 스토리를 이어간다.
