# 숏턴·엔터테인먼트형 CUA 스타일 “에이전트 vs 에이전트” 토론방 플랫폼 구현 심층 보고서

본 보고서는 **사용자가 토론 주제를 만들고(UGC), 서로 다른 에이전트(서로 다른 LLM 또는 서로 다른 프롬프트/페르소나 팩)가 짧게 대결(Agent‑vs‑Agent)** 하는 “엔터테인먼트 토론방” 플랫폼의 구현 가능성과 설계·사업성을 분석한다. CUA는 문맥에 따라 의미가 달라질 수 있으나, 본 보고서는 **‘숏턴 엔터 토론방’ 해석을 1순위**로 채택하고, 다른 해석(대화형/협업형/컴퓨터 사용형)이 UX·오케스트레이션·점수화에 미치는 영향도 비교한다. 이미 앱스토어/플레이스토어에는 “AI vs AI 토론 관전+평가” 앱이 존재하며, 오픈소스로는 **ELO·다중 심판·실시간 스트리밍**을 내세운 토론 아레나 구현체도 확인된다. citeturn5search0turn5search16turn4search3turn5search29 또한 “쌍대 비교 투표→레이팅/리더보드”는 대규모 아레나에서 검증된 운영 패턴이지만, **투표 조작(vote rigging)** 가능성이 학술적으로 제기된 만큼 게임화된 엔터 제품에서는 **조작 방지·신뢰 설계가 MVP 레벨 요구사항**이 된다. citeturn1search2turn0search15 국내 시장은 모바일 중심 게임 소비가 매우 크고, AI 엔터테인먼트 채팅 서비스가 높은 사용시간/이용자 지표를 제시하고 있어 “짧은 관전형 대결 콘텐츠”로 확장 실험을 할 토대는 존재한다. citeturn2search0turn2search1

## 정의

**가정(중요)**: 목표 예산·정확한 타깃 연령·출시 채널(모바일/웹) 미지정 → 본 보고서는 *국내 우선(domestic‑first), 모바일 퍼스트, 청소년 보호를 내장한 엔터테인먼트 서비스*를 기본 가정으로 한다.

### CUA(모호한 용어)의 내재적 정의 2–3가지와 본 보고서의 채택 정의

CUA는 통상 합의된 단일 확장이 존재하지 않으므로, 제품 설계에 영향을 주는 대표 해석 3가지를 명시한다. 이 중 1)번을 **주요(Primary)**로 채택한다.

1) **(Primary) 엔터테인먼트 숏턴 토론방형 CUA(본 보고서 채택)**  
- 정의(채택): “사용자가 주제·규칙·에이전트 조합을 생성하고, 복수 에이전트가 짧은 턴으로 대결한 뒤 관객 반응(투표/리액션/시청)을 기반으로 승패·랭킹을 만드는 UGC 게임형 플랫폼”  
- 평가 대상: *모델 코어 성능 벤치마킹*이 아니라 **프롬프트/페르소나 팩(연출·토론 스킬) 경쟁력**(‘누가 더 재밌게/설득력 있게 말하나’)  
- 유사 제품 신호: “두 AI가 어떤 주제든 배틀하고 사용자가 관전”류 앱이 다수 존재 citeturn5search0turn5search16turn4search3

2) **Conversational User Agent(대화형 사용자 에이전트) 해석**  
- 의미(가정): 1:1 챗봇/비서 중심. 목표는 문제 해결·만족도·개인화.  
- 본 제품에 미치는 영향: 대결(승패)보다 장기 관계/개인화 가치가 중심이 되어 “토론방은 부기능”이 되기 쉽다(UX가 분산).

3) **Computer‑Using Agent(컴퓨터 사용형 에이전트) 해석**  
- 의미(공식 용례): entity["company","OpenAI","ai company us"] 문맥에서 CUA는 GUI(버튼/메뉴/텍스트 필드 등)를 사람이 하듯 조작하도록 학습된 “Computer‑Using Agent”를 뜻한다. citeturn0search0turn0search4  
- 본 제품에 미치는 영향: 토론의 대상이 “말싸움”이 아니라 “작업 수행(웹/앱 조작) 대결”로 바뀌며, 점수화는 성공률·시간·오류복구·권한 준수 중심이 된다. citeturn0search4

### 시사점: “엔터 숏턴 Agent‑vs‑Agent”의 차별화 지점
- 앱스토어 사례는 이미 “AI vs AI 관전+평가”를 엔터 포맷으로 상품화했지만, 대체로 **(1) UGC 룸 생태계, (2) 프롬프트 팩/페르소나 마켓, (3) 조작 내성 있는 랭킹**을 강하게 내세우는 경우는 드물다(공식 설명 기준). citeturn5search0turn5search16turn4search3  
- 오픈소스 영역에서는 **ELO·다중 심판·실시간 스트리밍(SSE)**처럼 “게임화된 경쟁 운영” 요소가 명시적으로 구현되고 있어, 엔터 제품의 기술적 출발점(레퍼런스)은 충분하다. citeturn5search29  
- 다만 “투표 기반 랭킹”은 조작 가능성이 연구로 제기되므로, 엔터 제품은 **재미 설계만큼 ‘신뢰 설계’가 MVP 요구사항**이다. citeturn0search15

- entity["company","Microsoft","software company us"]의 AutoGen 문서는 “Multi‑Agent Debate”를 **멀티턴에서 에이전트들이 서로의 응답을 교환·개선**하는 설계 패턴으로 정의한다. citeturn1search0

- 또 다른 공식 문서/블로그에서도 “group chat orchestration(그룹챗 매니저가 턴을 조정)” 패턴이 토론/회의 시뮬레이션에 유용하다고 정리된다. citeturn1search20  

즉, 엔터 제품도 **(Orchestrator) 턴/룰 엔진 + (Agent Wrapper) 페르소나 팩 + (Judge Layer) 투표/간이 심판 + (Score Service) 레이팅/랭킹**으로 분해하면, 검증된 패턴 위에서 “숏턴·실시간” 요구사항을 얹는 문제로 환원된다.

### 엔터 숏턴에 최적화된 경량 점수화(제안)

아래 점수화는 “정답성”이 아닌 “프롬프트/토론 연출력”을 측정하기 위한 **엔터테인먼트 전용 설계(제안)**다.

1) **쌍대(또는 다자) 투표 기반 승패(Primary)**  
- 구조: 관객이 A/B/무승부/둘다별로를 택한다(“둘다별로”는 불만을 흡수해 이탈을 줄이기 위한 UX 장치로 권장).  
- 근거 패턴: Chatbot Arena는 쌍대 비교와 크라우드 투표를 기반으로 평가한다고 설명한다. citeturn1search2  

2) **프롬프트‑스킬(연출력) 보조 점수(Weighted Prompt‑Skill, Secondary)**  
- 목적: 방이 한산해 투표 데이터가 부족할 때 *점수 공백*을 메우고, “프롬프트 팩 품질”을 세부적으로 피드백하기 위함.  
- 구현: (a) 규칙 준수(길이/금칙/역할), (b) 반박 품질, (c) 임팩트/유머, (d) 논리 구조를 0~2의 작은 스케일로 채점.  
- 주의: LLM 심판은 편향(위치/장황함/자기향상)이 존재할 수 있으며, 이를 논문이 체계적으로 지적한다. citeturn4search1turn4academia33  

3) **Engagement 신호(시청/리액션/공유) — 낮은 가중치로만 사용(제안)**  
- 장점: 숏폼 엔터의 핵심은 시청 유지·공유이므로 제품 성장 지표와 정렬.  
- 위험: “자극적인 발언이 점수에 유리”해지는 Goodhart 효과가 생길 수 있어, (i) 안전 필터, (ii) 가중치 상한, (iii) 신고/제재와 묶어야 한다(제안).

4) **Time‑to‑Claim(훅이 나오는 속도)·Novelty(신선도) — 보조 피처(제안)**  
- Time‑to‑Claim: 첫 ‘주장(Claim)’ 또는 첫 ‘훅(재미 포인트)’까지의 토큰/초를 최소화(숏턴 연출).  
- Novelty: 반복 템플릿(“첫째, 둘째, 셋째…”) 과다 사용을 감점(임베딩 유사도/문장 패턴으로 간단히 근사).

### 권장 알고리즘/아키텍처

**오케스트레이터(룰 엔진)**  
- 입력: topic, 룰 프리셋(턴 수/발화 길이/시간 제한), 에이전트 팩(모델+프롬프트+금칙)  
- 처리: side 배정(찬/반), 턴 스케줄(ABAB 또는 라운드로빈), 컨텍스트 창(최근 N턴만), 스트리밍 출력

**에이전트 래퍼(Agent Wrapper / Persona Pack)**  
- Persona Pack은 “시스템 프롬프트 + 말투/금칙 + 길이 제한 + 특수 규칙(예: 첫 문장에 결론)”로 정의(제안).  
- 동일 모델이라도 Persona Pack이 다르면 “논쟁 스타일”이 달라지므로, 본 제품의 경쟁 단위는 **모델이 아니라 Persona Pack**이 된다(요구사항 정합).

**판정 레이어(Judge Layer)**  
- 기본: 관객 투표  
- 보조: LLM‑as‑a‑judge(경량) *단, 편향 완화 필요*  
  - 예: 위치 편향 완화를 위해 A/B 순서 교환 평가를 병행하는 방식은 LLM‑as‑a‑judge 연구에서 논의된다. citeturn4search1  
- “심판 모델도 비교·투표로 고를 수 있다”는 아이디어는 Judge Arena의 운영 컨셉과 유사하다. citeturn5search10turn7search2  

**레이팅/랭킹(Score Service)**  
- 실시간 업데이트가 중요한 엔터 제품은 **온라인 Elo**가 단순·저지연이다(제안).  
- 다만 Elo는 표본/순서에 민감할 수 있어, Chatbot Arena는 “데이터 셔플 후 다중 샘플링(bootstrap‑like)”로 변동성을 줄이는 기법을 도입했다고 설명한다. citeturn0search1  
- 리더보드가 조작될 수 있다는 연구(투표 조작)가 존재하므로, 랭킹이 곧 보상이 되는 구조라면 방어가 필수다. citeturn0search15  

### 최소 코드 스니펫(개념) — 숏턴 토론 진행 + 투표 기반 Elo 업데이트

```python
def run_short_turn_debate(topic, agents, turns=6, max_tokens=110):
    """
    agents: [AgentWrapper_A, AgentWrapper_B]
    returns: transcript = [(agent_id, text), ...]
    """
    transcript = []
    for t in range(turns):
        a = agents[t % 2]                # ABAB...
        side = "PRO" if (t % 2 == 0) else "CON"
        msg = a.generate(
            topic=topic,
            side=side,
            context=transcript[-4:],     # 짧은 컨텍스트(지연/비용 절감)
            max_tokens=max_tokens,
        )
        transcript.append((a.id, msg))
    return transcript

def vote_score(votes):
    """
    votes: {"A": n, "B": n, "draw": n, "both_bad": n}
    엔터 제품에서는 'both_bad'를 draw로 합치거나,
    match 품질 경고(노출/추천 감점)로 쓰는 것이 안전.
    """
    total = sum(votes.values()) or 1
    return (votes["A"] + 0.5 * votes["draw"]) / total

def elo_update(r_a, r_b, score_a, k=16):
    expected_a = 1 / (1 + 10 ** ((r_b - r_a) / 400))
    r_a2 = r_a + k * (score_a - expected_a)
    r_b2 = r_b + k * ((1 - score_a) - (1 - expected_a))
    return r_a2, r_b2
```

위 “쌍대 투표→레이팅” 패턴은 Chatbot Arena 계열에서 대규모로 사용된 운영 방식과 정렬되며, 변동성 완화(bootstrap) 같은 고급 기법도 공식 블로그에서 논의된다. citeturn1search2turn0search1  

### 점수화 방식별 장단점(엔터 숏턴 관점)

| 방식 | 장점 | 단점/리스크 | 추천 용도 |
|---|---|---|---|
| 투표 Only | 재미·공감 최적, 비용↓, 설명 쉬움 | 방이 한산하면 데이터 부족, 투표 조작 취약 | 기본값(초기) |
| LLM Judge Only | 즉시 결과(콜드스타트 완화) | 편향/환각, 비용·지연, “심판 신뢰” 이슈 | 보조/저가 모드 |
| 하이브리드(투표+LLM) | 데이터 부족 보정, 품질 피드백 가능 | 설계 복잡, 편향 보정 필요 | **권장**(LLM은 보조) |
| 참여지표 가중 | 숏폼 성장과 정렬 | 선정성 유인·안전 리스크 | 낮은 비율로만 |

LLM‑as‑a‑judge가 인간 선호와 높은 합의를 달성할 수 있다는 결과(80%+ 수준)와 동시에 편향이 존재한다는 지적이 함께 제시되므로, 엔터 제품에서 LLM 심판은 “주심”보다 “부심/보조”로 두는 편이 안전하다. citeturn4search1turn4search5turn4academia33

### 사업성·실현가능성(단·중·장기)
- **단기(6~12개월)**: 앱스토어/플레이스토어의 “AI vs AI 관전형 토론” 앱들과 오픈소스 토론 아레나 구현체가 존재하므로, Agent‑vs‑Agent 숏턴 토론방의 **기술적 구현 가능성은 높다**. citeturn5search0turn4search3turn5search29  
- **중기(12~24개월)**: 차별화는 모델 자체가 아니라 **Persona Pack 생태계(UGC 제작·개선·거래) + 신뢰 가능한 랭킹/조작 방지**에서 갈린다. 특히 투표 기반 시스템의 조작 가능성이 연구로 제기된 만큼, anti‑cheat가 경쟁력의 중심이 된다. citeturn0search15  
- **장기(24개월+)**: 청소년 안전·유해 콘텐츠·정서적 위해 이슈가 서비스 확장과 함께 커지며, 실제로 업계에서 미성년 오픈엔디드 채팅을 중단/제한하는 정책 변화가 발생했다. 장기적으로는 “성장”보다 “안전 거버넌스”가 먼저다. citeturn7search3turn7search0  

### 권장 전략 3가지(제품/시장/윤리) — 실행 가능한 형태로
1) **제품 전략: ‘모델 리더보드’가 아니라 ‘프롬프트 팩 리그’로 재정의**  
   - UI/랭킹/보상 단위를 “모델명”이 아니라 **Persona Pack**으로 고정하라(요구사항 정합).  
   - 프리셋(15초 쇼츠, 밈배틀, 팩트체크 동반전) 중심으로 “리믹스”를 설계하면 UGC 확산이 쉬워진다(제안).  

2) **시장 전략: 국내 모바일 엔터 구조에 맞춘 ‘스트리머‑퍼스트’ GTM**  
   - 국내는 모바일게임 매출 비중이 높아(59.3%) 숏턴 룸 콘텐츠가 모바일 엔터 흐름과 맞는다. citeturn2search0  
   - 스트리머/클립 확산 루프(투표 오버레이+자동 하이라이트)를 MVP에 내장해 “유입→공유→재방문”을 먼저 만든 뒤, 토큰/스폰서십을 얹는 순서를 권장한다(제안).  

3) **윤리/신뢰 전략: ‘anti‑cheat + safety’를 MVP 스펙으로 격상**  
   - 투표 기반 랭킹은 조작될 수 있으므로(연구 근거), 최소한의 신뢰 스택(가중치/이상탐지/레이트리밋)을 출시 기본값으로 두어야 한다. citeturn0search15  
   - Prompt Injection은 LLM 앱의 대표 리스크로 정리되어 있으므로, UGC 입력이 많은 엔터 제품일수록 “룰 템플릿 고정/입력 구조화/필터”를 필수로 설계하라. citeturn2search3
