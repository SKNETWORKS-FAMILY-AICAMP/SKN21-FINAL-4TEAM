## 1. 개요

### 이 문서의 목적

이 문서는 다음 세 가지 시나리오를 위한 로컬 환경 구성 방법을 설명합니다.

1. **초기 셋업** — `setup.sh` 한 번으로 venv, 의존성, 환경변수 파일까지 자동 구성
2. **테스트 실행** — `run-tests.sh` 한 번으로 DB/Redis 컨테이너 기동, pytest, vitest, E2E 테스트 실행, 컨테이너 정리까지 자동 처리
3. **풀스택 개발** — `local-dev.sh start` 한 번으로 DB, Redis 기동 후 마이그레이션과 시드 데이터 적재, 앱 서버까지 일괄 시작

### 기존 문제점

기존에는 `docker-compose.yml`에 정의된 7개 서비스(db, redis, backend, frontend, nginx, prometheus, grafana, langfuse)를 모두 올려야 했습니다. 테스트를 위해 Grafana, Langfuse 같은 무거운 서비스까지 기동해야 했고, 개발 DB와 테스트 DB가 같은 포트를 공유해 동시 실행이 불가능했으며, 신규 개발자가 `.env.test`를 직접 작성해야 했습니다.

### 해결 방법 요약

| 문제 | 해결책 |
|---|---|
| 신규 개발자 온보딩 시간 | `scripts/setup.sh` — venv, 의존성, 환경변수 파일 자동 생성 |
| `.env.test` 수동 작성 | `backend/.env.test.example` — 깃 추적 템플릿, 복사만 하면 완료 |
| 테스트용 경량 인프라 필요 | `docker-compose.test.yml` — DB/Redis만 포함 |
| 포트 충돌 (5432, 6379) | 테스트 포트를 5433/6380으로 분리 |
| 테스트 DB 초기화 속도 | `tmpfs` (RAM 기반 스토리지) 사용 |
| 자동화 | `run-tests.sh`, `local-dev.sh` 스크립트 |

## 3. 빠른 시작: setup.sh

**새 개발자는 클론 후 아래 명령어 하나만 실행하면 됩니다.**

```bash
bash scripts/setup.sh
```

DB, Redis, 마이그레이션, 시드 데이터, 백엔드, 프론트엔드까지 **모두 자동 처리**됩니다.

### 3.1 전체 실행 플로우

```
bash scripts/setup.sh
 │
 ├─ [환경 준비 — 최초 1회, 이후 스킵] ──────────────────────────
 ├─ Python 3.12+ 확인   → 미달 시 에러 종료
 ├─ Node.js 확인        → 없으면 경고 후 프론트엔드만 건너뜀
 ├─ Docker 확인         → 미실행 시 에러 종료 (DB 기동에 필수)
 ├─ backend/.venv 생성  → 이미 있으면 스킵
 ├─ pip install         → requirements.txt + requirements-dev.txt
 ├─ npm install         → 이미 있으면 스킵
 ├─ backend/.env 생성   → .env.example에서 복사 (이미 있으면 스킵)
 └─ backend/.env.test 생성 → .env.test.example에서 복사 (이미 있으면 스킵)
 │
 ├─ [인프라 기동] ────────────────────────────────────────────────
 ├─ docker compose up -d db redis  (docker-compose.yml, 포트 5432/6379)
 ├─ chatbot-db 헬스체크 대기
 ├─ chatbot-redis 헬스체크 대기
 ├─ alembic upgrade head
 └─ seed_data.py --env-file .env  (idempotent)
 │
 └─ [서버 기동 — 백그라운드] ────────────────────────────────────
     ├─ uvicorn app.main:app --reload  → logs/backend.log
     ├─ npm run dev                    → logs/frontend.log
     └─ 5초 후 PID 생존 여부 확인
```

완료 후 출력:

```
╔══════════════════════════════════════════════════════╗
║               개발 환경 준비 완료!                      ║
╚══════════════════════════════════════════════════════╝

  접근 주소
    Frontend:    http://localhost:3000
    Backend API: http://localhost:8000
    API 문서:    http://localhost:8000/docs

  테스트 계정
    admin     / Admin123!  (superadmin)
    moderator / Mod123!    (admin)
    user1     / User123!   (user)

  로그 확인
    tail -f logs/backend.log
    tail -f logs/frontend.log

  서버 종료
    bash scripts/setup.sh --stop

  테스트 실행
    bash scripts/run-tests.sh --backend-only
```

### 3.2 옵션

| 명령어 | 설명 |
|---|---|
| `bash scripts/setup.sh` | 셋업 + 서버 기동 (재실행 시 준비 단계 스킵, 서버만 재기동) |
| `bash scripts/setup.sh --stop` | 백엔드/프론트엔드 서버 종료 (DB/Redis 컨테이너는 유지) |
| `bash scripts/setup.sh --update-deps` | Python/Node 의존성 재설치 후 서버 기동 |

### 3.3 재실행 시 동작 (빠름)

두 번째 실행부터는 이미 완료된 단계를 스킵합니다.

| 단계 | 최초 실행 | 재실행 |
|---|---|---|
| venv 생성 + pip install | 수 분 | **스킵** |
| npm install | 수 분 | **스킵** |
| .env 파일 생성 | 생성 | **스킵** |
| Docker DB/Redis | 기동 | 이미 실행 중이면 그대로 사용 |
| 마이그레이션 | 적용 | 변경 없으면 즉시 완료 |
| 시드 데이터 | 삽입 | 이미 있으면 스킵 (idempotent) |
| 서버 기동 | 기동 | 기존 프로세스 종료 후 재기동 |

### 3.4 서버 로그 확인

백엔드/프론트엔드는 백그라운드로 실행되며 로그는 파일로 저장됩니다.

```bash
# 백엔드 로그 실시간 확인
tail -f logs/backend.log

# 프론트엔드 로그 실시간 확인
tail -f logs/frontend.log
```

### 3.5 .env.test.example 역할

`backend/.env.test.example`은 깃에 커밋되는 템플릿 파일입니다. 테스트 인프라의 비밀번호는 고정값이므로 팀원 누구나 API 키 없이 바로 사용할 수 있습니다.

```
저장소                         개발자 로컬
backend/.env.test.example  →  backend/.env.test  (setup.sh가 자동 복사)
          (깃 추적)                    (gitignore)
```

LLM 통합 테스트가 필요한 경우에만 `.env.test`에 API 키를 직접 추가합니다. 단위 테스트는 LLM 호출을 mock 처리하므로 API 키 없이 실행됩니다.

## 4. 사전 요구사항

> `setup.sh`가 아래 항목을 자동으로 확인하고 설정합니다. 수동으로 구성해야 하는 경우에만 이 섹션을 참고하십시오.

### 4.1 Docker Desktop

Docker Desktop이 설치되어 있어야 하며, 테스트 실행 전 반드시 **실행 중** 상태여야 합니다.

- 설치 링크: https://www.docker.com/products/docker-desktop/
- 최소 버전: Docker Desktop 4.x 이상 (Compose v2 내장)
- 확인 방법:

```bash
docker info
docker compose version
```

### 4.2 Python 가상환경 준비 (수동)

`setup.sh` 실행이 어려운 경우 직접 생성합니다.

```bash
# Windows (git-bash 또는 PowerShell)
cd backend
python -m venv .venv
.venv/Scripts/python.exe -m pip install -r requirements.txt
.venv/Scripts/python.exe -m pip install -r requirements-dev.txt

# Linux / macOS
cd backend
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt -r requirements-dev.txt
```

> **Windows 참고:** 스크립트는 `.venv/Scripts/python.exe`를 우선 탐색하고, 없으면 `.venv/bin/python`으로 폴백합니다. git-bash 환경에서 정상 동작합니다.

### 4.3 포트 사용 현황

개발 환경(`docker-compose.yml`)과 테스트 환경(`docker-compose.test.yml`)은 포트가 분리되어 있어 동시에 실행할 수 있습니다.

| 서비스 | 개발 환경 (docker-compose.yml) | 테스트 환경 (docker-compose.test.yml) |
|---|---|---|
| PostgreSQL | `127.0.0.1:5432` | `127.0.0.1:5433` |
| Redis | `127.0.0.1:6379` | `127.0.0.1:6380` |
| Backend API | `127.0.0.1:8000` | (컨테이너 없음, pytest가 직접 앱 임포트) |
| Frontend | `127.0.0.1:3000` | (컨테이너 없음, vitest가 직접 실행) |
| Nginx | `:80` | (없음) |
| Prometheus | `127.0.0.1:9090` | (없음) |
| Grafana | `127.0.0.1:3002` | (없음) |
| Langfuse | `127.0.0.1:3001` | (없음) |

> **설계 의도:** 테스트 인프라는 DB와 Redis만 띄웁니다. pytest는 FastAPI 앱을 직접 임포트(`ASGITransport`)하여 httpx로 호출하므로 백엔드 컨테이너가 불필요합니다.

## 5. 테스트 인프라 (docker-compose.test.yml)

### 5.1 구성 서비스

`docker-compose.test.yml`은 테스트에 필요한 최소 인프라만 포함합니다.

```yaml
services:
  test-db:      # pgvector/pgvector:pg16 — 포트 5433
  test-redis:   # redis:7-alpine — 포트 6380
```

| 서비스 | 컨테이너명 | 이미지 | 호스트 포트 |
|---|---|---|---|
| test-db | `chatbot-test-db` | `pgvector/pgvector:pg16` | `127.0.0.1:5433` |
| test-redis | `chatbot-test-redis` | `redis:7-alpine` | `127.0.0.1:6380` |

### 5.2 포트 분리 이유

개발 DB(5432)와 테스트 DB(5433)를 분리함으로써 개발 서버를 켜둔 채 테스트를 실행할 수 있습니다. 마찬가지로 Redis도 6379(개발) / 6380(테스트)로 분리됩니다.

```
개발 환경                    테스트 환경
chatbot-db:5432     동시실행  chatbot-test-db:5433
chatbot-redis:6379  가능      chatbot-test-redis:6380
```

### 5.3 tmpfs 사용 이유

테스트 DB는 디스크 볼륨 대신 RAM 기반 `tmpfs`를 사용합니다.

```yaml
# docker-compose.test.yml
test-db:
  tmpfs:
    - /var/lib/postgresql/data
```

- **속도:** 디스크 I/O 없이 RAM에서 직접 읽고 씁니다. 테스트용 INSERT/SELECT가 수 배 빠릅니다.
- **자동 초기화:** 컨테이너가 종료되면 데이터가 사라집니다. `docker compose down -v`로 볼륨을 지울 필요가 없습니다.
- **부작용 없음:** 개발 DB를 오염시키지 않습니다.

> **주의:** `tmpfs`이므로 컨테이너 재시작 시 데이터가 초기화됩니다. `--keep-infra` 옵션으로 컨테이너를 유지하더라도 재시작하면 데이터가 사라집니다.

### 5.4 Redis 테스트 설정

테스트 Redis는 RDB/AOF 영속성을 비활성화하고 메모리 64MB로 제한합니다.

```yaml
command: >
  redis-server
  --requirepass test_redis_pw
  --save ""          # RDB 스냅샷 비활성화
  --appendonly no    # AOF 비활성화 — 디스크 쓰기 없음
  --maxmemory 64mb
  --maxmemory-policy allkeys-lru
```

### 5.5 .env.test 환경변수 설명

`backend/.env.test`는 테스트 환경 전용 설정 파일입니다 (`backend/.env.test.example`에서 복사). 실제 API 키를 포함해서는 안 됩니다.

| 환경변수 | 값 | 설명 |
|---|---|---|
| `APP_ENV` | `development` | 앱 실행 환경 식별자 |
| `DEBUG` | `true` | 스택 트레이스 표시 활성화 |
| `DATABASE_URL` | `postgresql+asyncpg://...@127.0.0.1:5433/chatbot_test` | 비동기 DB 연결 (SQLAlchemy async) |
| `DATABASE_SYNC_URL` | `postgresql://...@127.0.0.1:5433/chatbot_test` | 동기 DB 연결 (Alembic 마이그레이션용) |
| `REDIS_URL` | `redis://:test_redis_pw@127.0.0.1:6380/0` | 테스트 Redis 연결 |
| `SECRET_KEY` | `test-secret-key-do-not-use-in-production` | JWT 서명 키 (테스트 전용, 운영 절대 사용 금지) |
| `DEBATE_ENABLED` | `true` | 대토론 기능 활성화 |
| `QUOTA_ENABLED` | `false` | 토큰 할당량 체크 비활성화 (테스트 편의) |
| `RATE_LIMIT_ENABLED` | `false` | API 속도 제한 비활성화 (테스트 편의) |
| `CREDIT_SYSTEM_ENABLED` | `false` | 크레딧 시스템 비활성화 (테스트 편의) |
| `PIPELINE_LAZY_LOAD` | `true` | 감정분석/임베딩 파이프라인을 실제 호출 시까지 로딩 지연 (시작 속도 개선) |
| `CORS_ORIGINS` | `["http://localhost:3000"]` | 허용 CORS 출처 |

## 6. 백엔드 테스트 실행

### 6.1 원커맨드 (run-tests.sh)

```bash
# 프로젝트 루트에서 실행
bash scripts/run-tests.sh [옵션]
```

#### 옵션 목록

| 옵션 | 설명 | 기본값 |
|---|---|---|
| _(옵션 없음)_ | 백엔드 + 프론트엔드 테스트 실행 | - |
| `--backend-only` | pytest (백엔드) 만 실행 | - |
| `--frontend-only` | vitest (프론트엔드) 만 실행 | - |
| `--e2e` | Playwright E2E 테스트 추가 실행 (단독 사용 가능) | `false` |
| `--all` | 백엔드 + 프론트엔드 + E2E 전체 실행 | - |
| `--no-infra` | 테스트 인프라 기동 건너뜀 (이미 실행 중인 경우) | `false` |
| `--keep-infra` | 테스트 완료 후 인프라 컨테이너 유지 | `false` |
| `--help` | 도움말 출력 | - |

#### 사용 예시

```bash
# 가장 일반적인 사용 — 백엔드 + 프론트엔드 테스트
bash scripts/run-tests.sh

# 백엔드 테스트만 빠르게
bash scripts/run-tests.sh --backend-only

# 인프라를 이미 켜둔 경우 (재기동 시간 절약)
bash scripts/run-tests.sh --backend-only --no-infra

# 테스트 후 DB를 유지해두고 데이터 확인
bash scripts/run-tests.sh --backend-only --keep-infra

# 전체 테스트 (E2E 포함) — CI에서 사용
```


### 6.2 수동 실행 (단계별)

자동 스크립트 없이 단계별로 실행하는 방법입니다. 개별 단계를 디버깅하거나 IDE에서 테스트를 직접 실행할 때 사용합니다.

**1단계 — 테스트 인프라 시작**

```bash
# 프로젝트 루트에서

# 헬스체크 확인

```

**2단계 — 테스트 DB에 마이그레이션 적용 (선택)**

단위 테스트는 `conftest.py`가 트랜잭션 롤백으로 DB를 격리하므로 마이그레이션이 필수는 아닙니다. 통합 테스트나 직접 DB 조회가 필요한 경우 실행합니다.

```bash

# Windows
DOTENV_PATH=".env.test" .venv/Scripts/python.exe -m alembic upgrade head

# Linux / macOS
DOTENV_PATH=".env.test" .venv/bin/python -m alembic upgrade head
```

**3단계 — pytest 실행**

```bash

# Windows

# Linux / macOS
DOTENV_PATH=".env.test" .venv/bin/python -m pytest tests/unit/ -v
```

**4단계 — 테스트 인프라 정리**

```bash

```

### 6.3 conftest.py DOTENV_PATH 지원

`backend/tests/conftest.py`는 최상단에서 `DOTENV_PATH` 환경변수를 읽어 지정된 `.env` 파일을 로드합니다.

```python
# backend/tests/conftest.py (상단 발췌)
_dotenv_path = os.environ.get("DOTENV_PATH")
if _dotenv_path:
    _env_file = Path(_dotenv_path)
    if _env_file.exists():
        with open(_env_file) as _f:
            for _line in _f:
                _line = _line.strip()
                if _line and not _line.startswith("#") and "=" in _line:
                    _key, _, _val = _line.partition("=")
                    os.environ.setdefault(_key.strip(), _val.strip())
```

`os.environ.setdefault`를 사용하므로 이미 설정된 환경변수는 덮어쓰지 않습니다. CI 환경에서 환경변수를 직접 주입하는 경우에도 `.env.test` 파일이 우선순위를 빼앗지 않습니다.

| 상황 | `DOTENV_PATH` 미설정 | `DOTENV_PATH=backend/.env.test` 설정 |
|---|---|---|
| DATABASE_URL | `backend/.env` 기본값 (5432) | `backend/.env.test` (5433) |
| REDIS_URL | `backend/.env` 기본값 (6379) | `backend/.env.test` (6380) |
| QUOTA_ENABLED | 설정 파일 값 | `false` (테스트 편의) |
| SECRET_KEY | 실제 키 | 테스트 전용 키 |

### 8.3 Idempotent 설계 설명

`seed_data.py`는 몇 번을 실행해도 안전합니다. 모든 엔티티에 고정 UUID를 부여하고, 실행 시 해당 UUID가 이미 존재하면 건너뜁니다.

```python
# 고정 UUID (항상 동일한 값)
SEED_IDS = {
    "superadmin": uuid.UUID("00000001-0000-0000-0000-000000000001"),
    "admin":      uuid.UUID("00000001-0000-0000-0000-000000000002"),
    ...
}

# 삽입 전 존재 여부 확인
exists = await session.get(User, data["id"])
if exists:
    continue  # 이미 있으면 스킵
session.add(User(**data))
```

이 방식의 장점은 다음과 같습니다.

- `local-dev.sh start`를 여러 번 실행해도 중복 데이터가 생기지 않습니다.
- 테스트 코드에서 고정 UUID로 시드 데이터를 참조할 수 있습니다.
- 새로운 시드 데이터를 추가해도 기존 데이터에 영향이 없습니다.

### 8.4 대상 DB 선택 방법

`--env-file` 옵션으로 개발 DB와 테스트 DB 중 하나를 선택합니다.

```bash
# 개발 DB에 시드 데이터 (기본값, docker-compose.yml DB 포트 5432)
cd backend
.venv/Scripts/python.exe ../scripts/seed_data.py --env-file .env

# 테스트 DB에 시드 데이터 (docker-compose.test.yml DB 포트 5433)
cd backend
.venv/Scripts/python.exe ../scripts/seed_data.py --env-file .env.test
```

`--env-file` 경로는 현재 디렉터리(CWD)와 `backend/` 디렉터리 순서로 탐색합니다.

### 문제 1. Docker가 실행 중이지 않을 때

```
[ERROR] Docker is not running. Please start Docker Desktop.
```

**해결:** Docker Desktop을 실행하고 고래 아이콘이 시스템 트레이에 표시될 때까지 기다린 뒤 재시도합니다.

```bash
docker info  # 성공 시 정상
```

### 문제 2. 포트 충돌 (5433 또는 6380이 이미 사용 중)

```
Error response from daemon: driver failed programming external connectivity:
Bind for 127.0.0.1:5433 failed: port is already allocated
```

**해결:** 이미 실행 중인 컨테이너를 확인하고 종료합니다.

```bash
# 포트를 사용 중인 프로세스 확인 (Windows)
netstat -ano | findstr :5433
netstat -ano | findstr :6380

# 또는 Docker 컨테이너 확인
docker ps -a | grep "5433\|6380"

# 테스트 인프라가 이미 기동 중인 경우 종료
docker compose -f docker-compose.test.yml down -v
```

### 문제 3. Python venv가 없을 때

```
[ERROR] .venv/Scripts/python.exe: No such file or directory
```

**해결 (권장):** `setup.sh`를 실행합니다.

```bash
bash scripts/setup.sh
```

**해결 (수동):**

```bash
cd backend
python -m venv .venv
.venv/Scripts/python.exe -m pip install -r requirements.txt -r requirements-dev.txt
```

### 문제 4. .env.test 파일이 없을 때

`backend/.env.test`는 `.gitignore` 대상이므로 저장소에 포함되지 않습니다.

**해결 (권장):** `setup.sh`를 실행합니다. `backend/.env.test.example`에서 자동 복사합니다.

```bash
bash scripts/setup.sh
```

**해결 (수동):**

```bash
cp backend/.env.test.example backend/.env.test
```

### 문제 5. 마이그레이션 실패 시

```
FAILED: Target database is not up to date.
```

**해결:** 마이그레이션 히스토리를 확인하고 현재 DB 상태와 코드 헤드를 비교합니다.

```bash
cd backend

# 현재 적용된 리비전 확인
DOTENV_PATH=".env.test" .venv/Scripts/python.exe -m alembic current

# 코드의 최신 헤드 확인
.venv/Scripts/python.exe -m alembic heads

# 마이그레이션 이력
.venv/Scripts/python.exe -m alembic history

# 강제 헤드까지 적용
DOTENV_PATH=".env.test" .venv/Scripts/python.exe -m alembic upgrade head
```

테스트 DB 전체를 초기화하려면 컨테이너를 내렸다 다시 올립니다. tmpfs이므로 자동으로 초기화됩니다.

```bash
docker compose -f docker-compose.test.yml down -v
docker compose -f docker-compose.test.yml up -d
```

### 문제 6. 시드 데이터 재생성 방법

시드 데이터를 완전히 다시 삽입하려면 테스트 DB를 초기화 후 마이그레이션을 재적용합니다.

```bash

.venv/Scripts/python.exe ../scripts/seed_data.py --env-file .env.test
```

개발 DB(`docker-compose.yml`)를 초기화하려면 `local-dev.sh reset`을 사용합니다.

```bash
bash scripts/local-dev.sh reset
```

## 10. CI/CD 연동 예시

GitHub Actions에서 `run-tests.sh`를 사용하는 워크플로우 예시입니다.

```yaml
# .github/workflows/test.yml
name: Test

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Set up Python 3.12
        uses: actions/setup-python@v5
        with:
          python-version: "3.12"

      - name: Create virtual environment
        run: |
          cd backend
          python -m venv .venv
          .venv/bin/pip install -r requirements.txt -r requirements-dev.txt

      - name: Set up Node.js 20
        uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"
          cache-dependency-path: frontend/package-lock.json

      - name: Install frontend dependencies
        run: cd frontend && npm ci

      - name: Create .env.test
        # .env.test.example이 저장소에 포함되어 있으므로 복사만 하면 됩니다.
        run: cp backend/.env.test.example backend/.env.test

      - name: Run all tests
        run: bash scripts/run-tests.sh --all

      - name: Upload test results
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: test-results
          path: |
            backend/.pytest_cache/
            frontend/playwright-report/
```

> **설계 의도:** CI 환경에서도 로컬과 동일한 스크립트(`run-tests.sh`)를 사용합니다. `.env.test.example`이 저장소에 포함되어 있어 `cp` 한 줄로 환경변수 준비가 끝납니다. "CI에서만 재현되는 오류"를 최소화하고, 개발자가 로컬에서 동일한 조건으로 검증할 수 있도록 합니다.
